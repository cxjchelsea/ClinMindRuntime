# 一、文档定位

ClinMindRuntime 不是单纯的 Python AI Demo，也不是普通后端 CRUD 系统。

它的长期定位是：

```text
以 Java / Spring Boot 为 Runtime 主控，
以 Provider / Agent / Tool 接入 AI 能力、知识资产、评估能力和治理能力，
以 PostgreSQL 等基础设施承载资产、评估、Trace、审计和平台治理，
以 Python 作为模型能力与实验能力的 Provider / Experiment 层，
以前端 Console 展示和治理 Runtime，而不是替代 Runtime。
```

全局技术栈必须同时满足：

```text
1. Runtime 主控稳定。
2. 医疗安全边界可控。
3. AI 能力可替换、可评估、可回滚。
4. 资产、评估、经验和治理可持续演进。
5. 不因过早引入复杂基础设施而破坏当前 Phase 主线。
```

# 二、总原则

```text
1. Java / Spring Boot 是 Runtime Core 主工程。
2. Python 是 AI Provider / Experiment 层，不是 Runtime 主控。
3. LLM / RAG / KG / embedding / MCP / Agent SDK 只能作为 Provider、Adapter、Agent 执行层参考或外部能力，不能绕过 RuntimeState、Runtime Validation、SafetyGate、DecisionBoundary。
4. PostgreSQL 是长期主数据库默认选型。
5. pgvector 是小规模向量能力默认优先方案，Milvus / Qdrant 后置。
6. KG-lite 优先用 PostgreSQL 表结构表达，Neo4j 后置。
7. React + TypeScript + Vite 是当前 Console 默认前端栈。
8. Docker Compose、Redis、监控和生产级权限后置到明确 Phase。
9. 所有技术引入都必须说明：为什么现在需要、服务哪个 Phase、是否破坏已有边界。
```

```text
  YAML / JSON / PostgreSQL

  Java Provider / Python FastAPI Provider / AgentRuntime / Tool Adapter / MCP Adapter

Storage / Integration 层
  PostgreSQL / Flyway / Redis / pgvector / 可选 Neo4j / 可选对象存储
  Runtime、Trace、Evaluation、Candidate、Review、Audit、Knowledge、Model metadata
```

# 五、Java 与 Python 的职责边界

## 5.1 Java 负责

```text
1. Runtime 主控。
2. RuntimeState 生命周期。
3. SafetyGate / Runtime Validation / DecisionBoundary。
4. Patient / Clinician 输出隔离。
5. RuntimeTrace 与 AuditLog。
6. Asset Package / Provider 接入边界。
7. EvaluationRunner 和结构化评估主流程。
8. Candidate / Review / Governance。
9. Console API / Safe DTO / AccessPolicy。
10. 后续 AgentExecutionLayer 的受控执行入口。
```

## 5.2 Python 负责

```text
1. LLM 调用实验。
2. CaseFrame 抽取实验。
3. embedding 生成。
4. RAG / reranker 实验。
5. 相似病例检索。
6. RuntimeTrace 离线分析。
7. 经验候选挖掘。
8. 模型训练 / 后训练实验。
9. AI Provider 服务化。
```

## 5.3 Python 不允许

```text
1. 不允许成为 Runtime 主控。
2. 不允许直接修改 RuntimeState。
3. 不允许绕过 Runtime Validation。
4. 不允许绕过 SafetyGate。
5. 不允许绕过 DecisionBoundary。
6. 不允许直接生成患者端最终诊断结论。
7. 不允许直接上线 CapabilityProfile、ExperienceUnit 或 TrainingDatasetVersion。
```

# 七、技术替换原则

```text
1. Runtime Core 技术不可轻易替换。
2. Provider 技术可替换，但接口契约不能随意变。
3. Storage 可以演进，但不能让数据库承担医疗主控。
4. 前端可以演进，但不能绕过 Safe Console API。
5. 模型可以替换，但必须有 ModelProviderVersion、EvaluationResult 和回滚能力。
6. RAG / GraphRAG 可以增强证据，但不能替代 EvidenceGraph。
7. MCP / Skills 可以连接能力，但不能替代 Runtime 主控。
```

---

# 八、禁止边界

```text
1. 不因为接入 Python 而把主流程迁到 Python。
2. 不因为接入 Agent SDK / LangGraph 而替换 RuntimeService。
3. 不因为接入 RAG 而让检索结果直接回答患者。
4. 不因为接入模型训练而扩大患者端输出权限。
5. 不因为接入前端而暴露 raw snapshot 或敏感字段。
6. 不因为接入数据库而删除 in-memory 回归路径。
7. 不因为接入 MCP 而允许外部工具写入 RuntimeState。
```

---

# 九、最终结论

ClinMindRuntime 的技术栈选择必须服务统一 Runtime 主链路。

```text
Java / Spring Boot 负责主控，
PostgreSQL / Audit / Console 负责治理底座，
Python / RAG / Model / MCP / Skills 负责受控能力扩展，
React Console 负责展示和治理，
Evaluation / Trace / Audit 负责证明和回滚。
```

任何技术接入都不能反过来替代 Runtime 主控。
