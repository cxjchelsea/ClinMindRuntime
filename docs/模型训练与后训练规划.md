# 模型训练与后训练规划

> 本文档定义 ClinMindRuntime 中模型训练、后训练、模型部署与 Runtime 的关系。  
> 本项目当前不是模型训练项目，但必须具备“哪些能力未来需要模型、训练数据从哪里来、训练结果如何作为 Provider 接入 Runtime、模型不能越过哪些安全边界”的长期规划。

---

# 一、文档定位

ClinMindRuntime 的主线不是直接训练一个“替代医生”的大模型，而是构建一个医疗 AI Runtime 与能力治理系统。

因此，模型训练在本项目中的定位是：

```text
模型训练 = 提升某些可替换 Provider 的能力
Runtime = 控制状态、安全、证据、边界、Trace 和最终输出责任
```

正确关系是：

```text
Training / Post-training
→ Model Provider
→ Structured Draft / Candidate / EvidenceRef
→ Runtime Validation
→ SafetyGate
→ DecisionBoundary
→ PatientOutput / ClinicianReport
→ RuntimeTrace
→ EvaluationResult
→ 再训练 / 资产更新 / 经验治理
```

错误关系是：

```text
训练一个模型
→ 模型直接回答患者
→ 绕过 Runtime / SafetyGate / DecisionBoundary
```

---

# 二、为什么当前 Phase 1–3 不训练模型

Phase 1–3 的核心任务是验证系统主线：

```text
Phase 1：Runtime 是否成立。
Phase 2：资产 Provider 是否成立。
Phase 3：能力是否能被评估和授权。
```

在这三个 P0 阶段过早训练模型会带来问题：

```text
1. 模型能力会掩盖 Runtime 架构是否成立。
2. 训练数据尚未由 EvaluationResult 和 RuntimeTrace 稳定沉淀。
3. 安全边界、患者端隔离、资产版本追踪还未完全进入评估闭环。
4. 个人项目很容易被训练成本拖垮。
5. 医疗场景不适合先训练黑箱模型再倒推安全边界。
```

因此当前策略是：

```text
先闭环，再评估，再基于评估结果决定训练什么。
```

# 三、模型能力分层

## 3.1 入口理解模型

对应 Runtime 模块：

```text
EntryAssessment
CaseFrameService
SymptomGroup detection
RiskHint detection
```

候选模型 / Provider：

```text
IntentClassifierProvider
WorkModeClassifierProvider
SymptomGroupClassifierProvider
RiskSignalClassifierProvider
CaseFrameExtractorProvider
```

输入：

```text
用户原始输入
基本信息
对话上下文摘要
```

输出必须是结构化对象：

```json
{
  "work_mode": "clinical_mode",
  "symptom_group": "chest_pain",
  "risk_hint": "possible_high_risk",
  "case_frame_draft": {
    "symptoms": ["胸闷", "出汗"],
    "duration": "unknown",
    "aggravating_factors": ["活动后"]
  },
  "confidence": 0.86
}
```

不得直接输出：

```text
你可能是心梗。
建议立即用某某药。
```

---

## 3.2 RAG / 证据检索模型

对应 Runtime 模块：

```text
MedicalKnowledgeProvider
EvidenceAssetProvider
EvidenceGraphService
QuestionTestPolicyService
```

候选能力：

```text
EmbeddingModelProvider
EvidenceRetrieverProvider
QueryRewriteProvider
EvidenceRerankerProvider
GuidelineChunkClassifier
```

训练 / 优化目标：

```text
1. 更准确召回指南证据。
2. 更好匹配症状群和诊断方向。
3. 更好区分强证据、弱证据和不适用证据。
4. 更好支持 EvidenceGraph 的结构化引用。
```

---

## 3.3 医生端推理与报告模型

对应 Runtime 模块：

```text
ClinicianReportService
DifferentialDiagnosisBoardService
EvidenceGraphService
```

候选能力：

```text
ClinicianReportDraftProvider
DdxExplanationProvider
EvidenceSummaryProvider
ReasoningFormatProvider
```

说明：

```text
医生端模型可以帮助生成报告草稿、解释候选诊断和总结证据，
但不能替代 DifferentialDiagnosisBoard 和 EvidenceGraph 的结构化判断。
```

---

## 3.4 患者安全表达模型

对应 Runtime 模块：

```text
PatientOutputService
DecisionBoundary
PatientBoundaryScorer
```

候选能力：

```text
PatientSafeRewriteProvider
DiagnosisLeakageDetector
MedicalAdviceRiskClassifier
UnsafeReassuranceDetector
```

训练目标：

```text
1. 避免患者端泄露诊断结论。
2. 避免高风险病例输出低风险安抚。
3. 用更自然的语言表达风险和就医建议。
4. 避免过度保守或过度自信。
```

注意：

```text
患者安全表达模型只能生成表达候选，最终允许输出什么仍由 DecisionBoundary 决定。
```

---

## 3.5 评估辅助模型

对应 Phase 3：

```text
EvaluationRunner
EvaluationScorer
EvaluationResult
```

候选能力：

```text
LlmJudgeScorer
FreeTextSafetyScorer
ExplanationQualityScorer
PatientToneScorer
```

定位：

```text
LLM-as-a-Judge 只能作为辅助评分器，不能作为唯一评估依据。
```

优先级：

```text
先实现 SafetyGateScorer / PatientBoundaryScorer / TraceCompletenessScorer 等确定性评分器，
再在 Phase 3-P1 引入 LlmJudgeScorer。
```

---

## 3.6 经验候选挖掘模型

对应 Phase 4：

```text
RuntimeTrace
EvaluationResult
SafetyViolation
RegressionFinding
ExperienceCandidate
ExperienceUnitAsset
```

候选能力：

```text
FailureClusterProvider
TraceSummarizationProvider
SimilarCaseRetriever
ExperienceCandidateMiner
CaseDifficultyClassifier
```

作用：

```text
从失败病例、医生修正、评估结果和 RuntimeTrace 中发现可复用经验候选。
```

注意：

```text
候选经验不能自动进入 Runtime，必须经过 Evaluation / Review / Governance。
```

# 四、训练数据来源

模型训练数据不能凭空生成，必须来自可追踪来源。

可用数据源：

```text
1. EvaluationCaseSet
2. EvaluationItemResult
3. EvaluationResult
4. MetricResult
5. SafetyViolation
6. PatientBoundaryViolation
7. RuntimeTrace
8. Doctor Feedback
9. Follow-up Outcome
10. ExperienceCandidate
11. Human-reviewed corrections
12. RAG evidence feedback
```

数据沉淀路径：

```text
Runtime Execution
→ RuntimeTrace
→ EvaluationRunner
→ EvaluationResult
→ RegressionFinding / SafetyViolation
→ TrainingExampleCandidate
→ Human Review / Rule-based Filter
→ TrainingDatasetVersion
→ Model Training / Post-training
→ ModelProviderVersion
→ Evaluation again
→ CapabilityProfileUpdateProposal
```

# 五、训练类型

## 5.1 传统监督训练

适用：

```text
意图识别
症状群分类
风险信号分类
安全泄露检测
```

候选模型：

```text
BERT / RoBERTa / DeBERTa 类小模型
轻量中文分类模型
医疗领域小模型
```

---

## 5.2 Embedding / Retrieval 优化

适用：

```text
证据检索
相似病例检索
经验候选检索
RuntimeTrace 错误聚类
```

可用方式：

```text
embedding model selection
reranker training
query rewrite tuning
hard negative mining
```

---

## 5.3 SFT / Instruction Tuning

适用：

```text
病例结构化抽取
医生端报告草稿
患者安全表达改写
错误分析总结
```

限制：

```text
SFT 模型只能作为 Provider，不能成为 Runtime 主控。
```

---

## 5.4 DPO / Preference Optimization

适用：

```text
患者表达风格偏好
医生端报告清晰度
高风险表达保守度
不安全回答抑制
```

前提：

```text
必须有人工或规则验证的 preference pair。
```

---

## 5.5 Distillation

适用：

```text
把大模型能力蒸馏到轻量分类器或抽取模型。
降低推理成本。
提高入口模块稳定性。
```

# 六、模型部署方式

模型不直接部署在 Runtime Core 内部。

推荐部署结构：

```text
Java Runtime Core
→ Provider Interface
→ Python AI Provider / Java AI Provider
→ Model Runtime / LLM API / Embedding Service
```

Python Provider 示例：

```text
clinmind-ai-provider/
  app/
    routers/
      intent.py
      caseframe.py
      embedding.py
      rag.py
      safety_expression.py
      judge.py
      experience_mining.py
    services/
      model_registry.py
      llm_client.py
      embedding_service.py
      retrieval_service.py
      inference_service.py
```

候选接口：

```text
POST /models/intent-classify
POST /models/extract-case-frame
POST /models/normalize-symptom
POST /models/risk-signal-classify
POST /models/generate-embedding
POST /models/retrieve-evidence
POST /models/rewrite-patient-safe
POST /models/judge-output-safety
POST /models/mine-experience-candidates
```

# 七、模型注册与版本治理

后续需要模型注册结构：

```text
ModelProviderMetadata
- model_provider_id
- model_name
- model_type
- version
- task_type
- owner
- training_dataset_version
- evaluation_run_id
- status
- created_at
- risk_critical
```

模型状态：

```text
DRAFT
EVALUATING
APPROVED
DEPRECATED
DISABLED
```

模型上线规则：

```text
1. 必须有 EvaluationResult。
2. 必须通过安全与患者边界指标。
3. 必须记录训练数据版本。
4. 必须记录模型版本。
5. 必须可以回滚。
6. 风险关键模型必须 fail-closed。
```

# 八、与 CapabilityProfile 的关系

模型能力不能自动扩大输出权限。

正确关系：

```text
ModelProviderVersion
→ EvaluationResult
→ CapabilityProfileUpdateProposal
→ Review / Governance
→ CapabilityProfileAsset
→ DecisionBoundary
```

错误关系：

```text
模型表现看起来不错
→ 直接允许患者端输出更多诊断信息
```

# 九、按 Phase 的训练接入规划

## Phase 1-P0

```text
不训练模型。
用规则 / YAML / mock 建立 Runtime 主线。
EntryAssessment 可以先用规则和关键词。
```

## Phase 2-P0

```text
不训练模型。
建立 Provider 和 Asset Package，为后续模型 Provider 接入预留边界。
```

## Phase 3-P0

```text
不训练模型。
建立 EvaluationCaseSet、EvaluationRunner、Scorer、EvaluationResult。
先用确定性评估器发现能力短板。
```

## Phase 3-P1

```text
可开始沉淀 TrainingExampleCandidate。
可引入 LlmJudgeScorer 作为辅助评分器。
可用 Python notebook 做离线错误分析和指标可视化。
```

## Phase 4-P0

```text
引入 Python offline analysis。
引入 embedding / similar case retrieval experiment。
引入 ExperienceCandidate mining。
可训练或微调小型分类器：intent、symptom_group、risk_signal、patient_boundary_detector。
```

## Phase 4-P1

```text
引入 RAG EvidenceProvider 原型。
引入 reranker / query rewrite 实验。
引入 GraphRAG / KG-lite evidence experiment。
```

## Phase 5-P0

```text
服务化 Python AI Provider。
引入 ModelProviderMetadata / ModelRegistry。
引入 PostgreSQL / pgvector 持久化训练数据、评估结果、模型版本。
通过 Docker Compose 部署 Java Runtime、Python Provider、PostgreSQL、Redis、Frontend。
```

## Phase 5-P1 或后置

```text
考虑 SFT / DPO / Distillation。
考虑更完整模型管理后台。
考虑模型灰度发布、回滚、A/B 评估。
考虑更强模型监控和漂移检测。
```

# 十、禁止边界

```text
1. 不训练一个直接面向患者输出诊断的大模型。
2. 不让模型直接修改 RuntimeState。
3. 不让模型绕过 SafetyGate。
4. 不让模型绕过 DecisionBoundary。
5. 不让模型直接上线 CapabilityProfile。
6. 不把未审核的 RuntimeTrace 自动转为训练数据。
7. 不把 LLM-as-a-Judge 作为唯一安全评估依据。
8. 不把模型训练作为 Phase 3-P0 的必要内容。
```

# 十一、与“小儿方式四块能力”的对应关系

| 小儿方式能力 | ClinMindRuntime 对应 | 当前状态 |
|---|---|---|
| Agent 推理 | Runtime Core / Provider / DecisionBoundary | Phase 1/2 已完成 P0 |
| RAG 知识库 | EvidenceProvider / RAG / GraphRAG / pgvector | 预留，Phase 4/5 接入 |
| 后训练 | Evaluation-driven Training / Model Provider / TrainingDatasetVersion | 本文档规划，Phase 3-P1 起沉淀数据 |
| 部署 | Java Runtime + Python Provider + PostgreSQL + Docker Compose | 全局技术栈已规划，Phase 5 接入 |

---

# 十二、最终结论

```text
ClinMindRuntime 当前不是模型训练项目，但必须具备模型训练与后训练演进路线。

短期先用规则、YAML、结构化 Provider 和 Evaluation 建立闭环。
中期用 EvaluationResult 和 RuntimeTrace 生成训练数据。
后期将意图识别、病例抽取、症状归一化、证据检索、安全表达、经验挖掘等能力训练为可替换 Provider。
所有模型只能输出结构化候选，不能绕过 Runtime、安全门和输出边界。
```
