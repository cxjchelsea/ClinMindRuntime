# 架构文档缺口审查清单

> 本文档用于审查 ClinMindRuntime 当前架构文档体系的覆盖情况、缺口、优先级和后续补文档顺序。  
> 本文档不是新的系统设计，也不是要求立即实现所有能力，而是避免继续以“发现一个缺口就补一个文档”的方式被动扩张。

---

# 一、为什么需要这份文档

此前文档体系已经覆盖了 Runtime、共享能力资产、Evaluation、全局技术栈、AI 前沿技术、模型训练与后训练、技术实现总方案等主干内容。

但这些文档主要是按问题逐步补齐的：

```text
发现技术栈粗 → 补全局技术栈
发现 AI 前沿技术缺失 → 补 AI 前沿技术规划
发现训练路线缺失 → 补模型训练与后训练规划
发现技术落地缺少总方案 → 补技术实现总方案
发现知识库搭建没说明 → 暴露知识库 / RAG 构建规划缺口
```

这说明当前文档体系已经形成主干，但还没有经过系统性覆盖审查。

因此，本文件的目标是：

```text
1. 明确哪些能力域已经有足够文档支撑。
2. 明确哪些能力域只是分散提及，仍缺专项规划。
3. 明确哪些缺口必须优先补，哪些可以后置。
4. 避免把“文档越来越多”误认为“文档体系已经完整”。
5. 防止后续实现时因为文档缺口导致架构漂移。
```

# 三、覆盖度判定标准

本文档使用以下覆盖状态：

| 状态 | 含义 |
|---|---|
| 充分 | 已有总体设计、阶段边界、实现方案或 Phase 详细设计，当前不需要新增专项文档 |
| 基本覆盖 | 已有规划，但后续实现前可能需要补具体规格 |
| 分散覆盖 | 多个文档中提到，但缺少统一专项规划 |
| 明显缺口 | 关键能力域缺少专项规划，后续实现容易跑偏 |
| 后置缺口 | 不是当前阶段必需，但 Phase 5 或产品化前需要补 |

优先级：

| 优先级 | 含义 |
|---|---|
| P0 | 近期应补；不一定阻塞 Phase3-P0 代码，但会影响主线理解或后续关键能力 |
| P1 | Phase4 / Phase5 前应补 |
| P2 | 平台化、产品化、部署化前补 |
| 暂不补 | 当前已有文档足够，或能力不属于当前项目主线 |

# 五、当前最关键缺口

## 5.1 P0：医学知识库与 RAG 构建规划

这是当前最明显的文档缺口。

原因：

```text
1. 项目多次提到 Knowledge Context、RAG Evidence、KG-lite、GraphRAG、EvidenceProvider。
2. Phase2 已经做了静态 evidence refs 和 provider 边界。
3. Phase4-P1 / Phase5 计划接入 RAG / GraphRAG / pgvector。
4. 但当前还没有说明知识库如何搭建、如何入库、如何切分、如何版本化、如何进入 EvidenceGraph。
```

如果不补这个文档，后续很容易把知识库做成普通 RAG：

```text
文档切块 → embedding → 检索 → 拼 prompt → LLM 回答
```

这会绕过 ClinMindRuntime 的核心边界。

正确路线应是：

```text
RuntimeState / CaseFrame
→ KnowledgeContextService
→ EvidenceAssetProvider
→ RAG / KG-lite / GraphRAG
→ EvidenceRef / EvidenceCandidate
→ EvidenceGraph
→ SafetyGate / QuestionTestPolicy / DecisionBoundary
→ PatientOutput / ClinicianReport
```

建议新增：

```text
docs/医学知识库与RAG构建规划.md
```

该文档应覆盖：

```text
1. 医学知识库在 Runtime 中的定位。
2. 知识资产分类：规则、路径、证据文档、KG-lite、经验、评估病例。
3. 文档入库流程。
4. chunk 切分与元数据规范。
5. EvidenceRef / EvidenceCandidate 数据结构。
6. RAG Provider 输入输出。
7. KG-lite 与 GraphRAG 的边界。
8. PostgreSQL / pgvector / Neo4j / Milvus 接入阶段。
9. 知识审核、发布、版本、回滚。
10. 知识库评估指标。
11. 各 Phase 接入计划。
```

## 5.2 P1：数据安全与合规边界规划

医疗项目即使是个人作品，也需要说明安全边界。

当前已有：

```text
患者端 / 医生端输出隔离
debug API 不面向 patient-facing client
数据脱敏和访问审计后置
```

但缺少统一规划：

```text
1. 哪些数据是敏感数据。
2. RuntimeTrace 中哪些字段需要脱敏。
3. EvaluationCaseSet 是否可包含真实病例。
4. Doctor Feedback / Follow-up Outcome 如何匿名化。
5. debug/internal API 如何避免暴露患者数据。
6. 日志、审计、导出、训练数据如何脱敏。
```

建议新增：

```text
docs/数据安全与合规边界规划.md
```

优先级：P1。

# 八、当前文档体系真实状态

当前真实状态不是“已经完美”，而是：

```text
主干架构文档体系已经建立，
Phase1 / Phase2 / Phase3 的主线基本清晰，
全局技术栈、AI 前沿技术、模型训练和技术实现边界已经有规划，
但若按完整医疗 AI 系统审查，仍存在若干专项规划缺口。
```

更准确的覆盖判断：

```text
主干设计覆盖度：较高。
当前 Phase3 实现所需文档：基本足够。
Phase4/5 能力接入所需文档：仍需补充。

数据安全、测试 CI、部署、前端平台：后续补齐。
```


```text
Phase3 开始前：检查 Evaluation 文档是否足够。
Phase4 开始前：检查知识库 / RAG、经验进化、训练数据、数据安全是否足够。
Phase5 开始前：检查数据库、前端、部署、权限、审计、模型治理是否足够。
```

# 十、最终结论

ClinMindRuntime 当前文档体系已经具备主干，但不能说已经完全闭环。

当前结论：

```text
1. 可以继续推进 Phase3-P0-A 代码实现。
2. 近期应补 docs/医学知识库与RAG构建规划.md。
3. Phase4/5 前应补数据安全、测试 CI、数据库持久化、前端平台、部署运维等专项规划。
4. 以后不再凭感觉判断“文档是否完整”，而以本缺口审查清单为依据。
```
