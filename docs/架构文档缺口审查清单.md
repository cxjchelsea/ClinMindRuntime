# 三、覆盖度判定标准

| 状态 | 含义 |
|---|---|
| 充分 | 已有总体设计、阶段边界、技术实现方案或 Phase 详细设计，当前不需要新增专项规划 |
| 规划级覆盖 | 已有专项规划，但进入实现前仍需要实现规格 / API / 表结构 / 任务清单 |
| 明显缺口 | 关键能力域缺少专项规划，后续实现容易跑偏 |
| 后置缺口 | 当前非必要，但产品化前需要补 |
| 已冻结 | 当前阶段目标已达成，后续只允许 bug fix / 测试补强 / 文档同步 |
| 当前基线 | 当前 Phase 实现必须遵守的设计文档 |
| implementation ready | 当前 Phase 设计已完成，可以进入任务清单中的下一项实现 |

# 五、当前仍需注意的点

## 5.1 没有明显 P1 文档缺口

当前不需要继续补 Phase5-P0 总体规划文档。

当前重点是：

```text
1. 保持 Phase5-P0 冻结边界。
2. 规划 Phase5-P1（最小 Console / RBAC 强化）。
3. 保持 in-memory 与 postgres 专项回归可复现。
4. 不跳到 RAG / 模型训练 / 完整审核平台。
```

## 5.2 当前不应实现的后置能力

以下能力仍然后置：

```text
真实 RAG / GraphRAG
Python AI Provider
前端 Console
模型训练 / 后训练
MCP / LangGraph / Agent SDK
完整 RBAC / 登录 / 多租户
正式医生审核平台
正式 ExperienceMemory 生效机制
TrainingDatasetVersion 发布机制
```

## 5.3 Phase5-P0 推荐主题

```text
PostgreSQL 持久化 + Repository 双实现 + AuditLog 最小治理
```

推荐链路：

```text
Service
→ Store Interface
→ InMemory / PostgreSQL implementation
→ AuditLog
→ Persistence health / E2E
```
