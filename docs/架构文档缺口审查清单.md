# 三、覆盖度判定标准

| 状态 | 含义 |
|---|---|
| 充分 | 已有总体设计、阶段边界、技术实现方案或 Phase 详细设计，当前不需要新增专项规划 |
| 规划级覆盖 | 已有专项规划，但进入实现前仍需要实现规格 / API / 表结构 / 任务清单 |
| 明显缺口 | 关键能力域缺少专项规划，后续实现容易跑偏 |
| 后置缺口 | 当前非必要，但产品化前需要补 |
| 已冻结 | 当前阶段目标已达成，后续只允许 bug fix / 测试补强 / 文档同步 |
| 当前基线 | 当前 Phase 实现必须遵守的设计文档 |
| 已归档 | Phase 目标已达成，任务清单与验收记录作为历史依据 |
| implementation ready | 当前 Phase 设计已完成，可以进入任务清单中的下一项实现 |

# 五、当前仍需注意的点

## 5.1 没有明显 P2 后文档缺口

当前不需要为「已完成的最小 Console MVP」再补总体规划文档。

当前重点是：

```text
1. 保持 mvn test 与 console-web npm run test / build 全绿。
2. 文档与代码状态一致（尤其技术实现总方案、项目展示导读、本文件）。
3. 后置能力须单独立项，不得破坏 P1/P2 冻结边界。
4. 不跳到 RAG / 模型训练 / 完整审核平台 / 正式 OAuth。
```

# 六、后续更新规则

以后每新增或修改一个架构文档，必须回到本文件更新：

```text
1. 对应能力域的覆盖状态。
2. 是否需要实现规格。
3. 是否影响当前阶段判断。
4. 是否需要加入 AI_IMPLEMENTATION_SKILL.md 权威文档优先级。
5. 是否需要同步 docs/README.md 与 ClinMindRuntime技术实现总方案.md。
```

每次声称“文档体系完整”前，必须检查本文件。
