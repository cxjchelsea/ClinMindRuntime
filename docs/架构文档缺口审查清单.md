因此，当前文档体系从“主干文档已建立，但专项规划存在明显缺口”，更新为：

```text
主干架构文档已建立。
Phase 1 / Phase 2 / Phase 3 的当前实现所需文档基本充分。
Phase 4 / Phase 5 所需的关键专项规划已有规划级文档。
后续进入具体 Phase 前，仍需要把对应规划升级为实现规格、API 设计、表结构设计或任务清单。
```

注意：

```text
规划级覆盖 ≠ 已经实现。
专项规划补齐 ≠ 可以提前实现 Phase 4/5。
```

# 三、覆盖度判定标准

| 状态 | 含义 |
|---|---|
| 充分 | 已有总体设计、阶段边界、技术实现方案或 Phase 详细设计，当前不需要新增专项规划 |
| 规划级覆盖 | 已有专项规划，但进入实现前仍需要实现规格 / API / 表结构 / 任务清单 |
| 分散覆盖 | 多个文档中提到，但缺少统一专项规划 |
| 明显缺口 | 关键能力域缺少专项规划，后续实现容易跑偏 |
| 后置缺口 | 当前非必要，但产品化前需要补 |

# 五、当前仍需注意的点

## 5.1 没有明显 P0 文档缺口

当前 Phase3-P0 代码实现所需文档已经足够，包括：

```text
Phase3_开发任务清单
Phase3_训练与评估闭环_实现规格
Phase3_评估数据结构设计
Phase3_病例集与考试流程设计
Phase3_Runtime评估接入设计
Phase3_CapabilityProfile更新机制设计
Phase3_API与测试设计
ClinMindRuntime技术实现总方案
测试与CI总方案
```

当前最优下一步仍然是：

```text
Phase3-P0-A：Evaluation 数据结构
```

## 5.2 规划文档不能变成提前实现理由

虽然已经补齐六份专项规划，但以下能力仍然后置：

```text
真实 RAG / GraphRAG
Python AI Provider
PostgreSQL / Redis / pgvector
前端 Console
模型训练 / 后训练
MCP / LangGraph / Agent SDK
完整权限 / 审计 / 部署体系
```

## 5.3 进入新 Phase 前仍要升级文档

```text
进入 Phase4 前：补 Phase4 经验进化与训练数据沉淀详细设计。
进入 Phase4-P1 前：补 RAG Provider / GraphRAG 实现规格。
进入 Phase5 前：补数据库 schema、Console API、RBAC、部署实现规格。
```

# 六、后续更新规则

以后每新增或修改一个架构文档，必须回到本文件更新：

```text
1. 对应能力域的覆盖状态。
2. 是否从“明显缺口”变为“规划级覆盖”。
3. 是否还需要实现规格。
4. 是否需要加入 AI_IMPLEMENTATION_SKILL.md 权威文档优先级。
```

每次声称“文档体系完整”前，必须检查本文件。
