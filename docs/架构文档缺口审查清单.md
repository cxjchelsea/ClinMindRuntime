# 三、覆盖度判定标准

| 状态 | 含义 |
|---|---|
| 充分 | 已有总体设计、阶段边界、技术实现方案或 Phase 详细设计，当前不需要新增专项规划 |
| 规划级覆盖 | 已有专项规划，但进入实现前仍需要实现规格 / API / 表结构 / 任务清单 |
| 分散覆盖 | 多个文档中提到，但缺少统一专项规划 |
| 明显缺口 | 关键能力域缺少专项规划，后续实现容易跑偏 |
| 后置缺口 | 当前非必要，但产品化前需要补 |
| freeze cleanup | 当前不是补规划，而是修正状态、整理导航、做小型一致性清理 |

# 五、当前仍需注意的点

## 5.1 没有明显 P0 文档缺口

当前不需要继续补总体规划文档。

当前重点是：

```text
1. Phase3-P0 freeze cleanup。
2. 小型一致性修复。
3. 测试和人工验收保持可复现。
4. 项目展示入口继续完善。
```

## 5.2 当前不应进入 Phase 4

以下能力仍然后置：

```text
真实 RAG / GraphRAG
Python AI Provider
PostgreSQL / Redis / pgvector
前端 Console
模型训练 / 后训练
MCP / LangGraph / Agent SDK
完整权限 / 审计 / 部署体系
```

## 5.3 freeze cleanup 建议项

```text
1. 校验 EvaluationRunConfig.caseSetVersion 与 manifest.version 一致。
2. 优化 notApplicable 指标语义，避免未配置指标被当作满分误导。
3. 修正 Evaluation item not found 的错误码。
4. 更新 pom.xml description。
5. 补根目录 README。
6. 运行测试或记录未运行原因。
```

## 5.4 进入新 Phase 前仍要升级文档

```text
进入 Phase4 前：补 Phase4 经验进化与训练数据沉淀详细设计。
进入 Phase4-P1 前：补 RAG Provider / GraphRAG 实现规格。
进入 Phase5 前：补数据库 schema、Console API、RBAC、部署实现规格。
```

# 六、后续更新规则

以后每新增或修改一个架构文档，必须回到本文件更新：

```text
1. 对应能力域的覆盖状态。
2. 是否需要实现规格。
3. 是否影响当前阶段判断。
4. 是否需要加入 AI_IMPLEMENTATION_SKILL.md 权威文档优先级。
5. 是否需要同步 docs/README.md。
```

每次声称“文档体系完整”前，必须检查本文件。
