> 上位总设计：`docs/ClinMindRuntime完整系统设计.md`

> 对应能力域 / 架构层：Storage / Integration 层；平台治理层；部署、运维、发布与回滚能力

# 二、当前阶段部署方式

Phase 1–3 默认：

```text
mvn test
mvn package
java -jar target/*.jar
Postman / curl 手工验收
JUnit 自动测试
```

不引入：

```text
Docker Compose
PostgreSQL
Redis
Python Provider
Frontend
Kubernetes
复杂 CI/CD
```

原因：

```text
当前重点是 Runtime、Asset、Evaluation 主线，而不是运维复杂度。
```

# 三、Phase 5 目标部署拓扑

Phase 5 推荐 Docker Compose 多服务：

```text
clinmind-runtime      Java Spring Boot Runtime Core
clinmind-ai-provider  Python FastAPI Provider
clinmind-postgres     PostgreSQL + pgvector
clinmind-redis        Redis
clinmind-frontend     React + TypeScript Console
```

关系：

```text
Frontend
→ Java Runtime API
→ PostgreSQL / Redis
→ Python AI Provider
→ External LLM / Model / RAG services
```

# 六、启动顺序

推荐顺序：

```text
1. PostgreSQL
2. Redis
3. Python AI Provider
4. Java Runtime
5. Frontend
```

Java Runtime 启动前检查：

```text
数据库连接可用。
Redis 可用。
资产包可加载。
Python Provider 可选：不可用时应进入降级模式，而不是影响核心 Runtime。
```

# 十二、禁止提前引入

```text
1. 不在 Phase 3-P0 引入 Docker Compose 作为必要条件。
2. 不提前上 Kubernetes。
3. 不提前做复杂微服务治理。
4. 不让部署结构改变 Runtime 主控边界。
5. 不让 Python Provider 变成主服务入口。
```

---

# 十三、最终结论

部署路线必须服务架构主线：

```text
本地 Java Runtime
→ Docker Compose 多服务
→ PostgreSQL / Redis / Python Provider / Frontend
→ 后置监控和云原生部署
```

Phase 1–3 不应被部署复杂度拖慢；Phase 5 再进入平台化部署。
