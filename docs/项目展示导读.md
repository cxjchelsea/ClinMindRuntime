# 一、项目一句话介绍

ClinMindRuntime 是一个面向医疗问诊场景的受控 AI Runtime 原型。

它的重点不是让大模型直接生成最终回答，而是让 AI 能力在一个可控、可评估、可追踪、可回滚、可治理的 Runtime 中运行。

```text
普通 RAG / Agent：输入 → 检索 / 工具 → LLM → 输出
ClinMindRuntime：输入 → RuntimeState → SafetyGate → EvidenceGraph → DecisionBoundary → 分角色输出 → RuntimeTrace → Evaluation → Candidate → Review → Persistence / Audit → Console Governance
```

---

# 二、项目要解决的问题

在高风险问诊场景中，AI 系统不能只追求回答自然，还需要处理：

```text
1. 高风险信号不能被忽略。
2. 患者端不能看到内部推理和医生端信息。
3. 知识、规则、经验、模型需要有版本和来源。
4. 每次输出需要能追踪依据。
5. 能力需要通过病例集和指标评估。
6. 外部模型、RAG、MCP 等能力不能绕过 Runtime 主控。
7. 运行结果即使有价值，也只能先变成候选，不能自动生效。
8. 核心治理对象需要持久化、审计和恢复能力。
9. 已持久化对象需要通过最小 Console API 被安全查询和权限控制。
```

# 五、当前最值得展示的能力

## 5.1 受控 Runtime，而不是普通问答链

一次交互会进入 Runtime 实例，并维护状态、证据、安全门、输出边界和 Trace。

## 5.2 患者端 / 医生端输出隔离

患者端只返回安全表达，医生端可以看到更多内部辅助信息。

## 5.3 Asset Package 版本治理

系统会绑定 asset_package_id 和 asset_package_version，版本不匹配时进入安全处理，而不是静默继续。

## 5.4 EvaluationScorer 策略体系

当前已有评分维度：

```text
EntryAssessmentScorer
SafetyGateScorer
PatientBoundaryScorer
DdxCoverageScorer
NextActionScorer
TraceCompletenessScorer
AssetVersionTraceScorer
```

## 5.5 Candidate 候选沉淀与 Review 边界

系统可以把失败、风险和回归结果转化为：

```text
ExperienceCandidate
TrainingExampleCandidate
```

Candidate 经过脱敏和 SourceRef 强校验，并可记录 review 决策；但即使 review_status 变成 `APPROVED`，也不会自动上线经验、不会自动进入训练集、不会自动修改资产包或 CapabilityProfile。

## 5.6 PostgreSQL-backed governance runtime

Phase 5-P0 已经把核心治理对象推进到 PostgreSQL 持久化、Repository 双实现和 AuditLog 最小治理。

## 5.7 最小 Console 与访问治理（P1 + P2 已交付）

Phase 5-P1 已交付最小 Console API：ActorContext、RBAC-lite、Safe Console DTO、Audit Center 查询。

Phase 5-P2 已交付 `console-web/` 最小治理界面：Runtime / Evaluation / Candidate / Review Queue / Audit Center 五页，含 Debug Context 与 review 表单。这不是完整产品化 Training Center。

## 5.8 下一阶段展示重点：受控 Agent Runtime

Phase 6-P0 建立后，展示重点应增加：

```text
AgentRegistry
AgentRuntime
AgentPolicy
AgentProposal
AgentProposalValidator
Runtime Validation
AgentTrace
InquiryPlanningAgent
```

展示目标不是证明“Agent 自己会看病”，而是证明：

```text
Agent 能在 Runtime 授权下生成可校验 Proposal，
Runtime 能决定采纳、部分采纳、拒绝或降级，
Agent 执行过程能进入 Trace / Evaluation / Audit。
```

# 六、5 分钟 Demo 路径

### 1. 启动

```powershell
set JAVA_HOME=D:\cxj\software\jdk21
mvn -DskipTests package
java -jar target\clinmind-runtime-0.1.0-SNAPSHOT.jar
```

### 2. 跑一次 Runtime（高风险胸痛）

`POST /api/v1/runtime/start` — 输入「胸口闷，活动后更明显，出汗」

期望：`safety_gate_triggered`；患者输出含「风险信号」；响应中 `differential_board` 为 `null`。

### 3. 跑一次 Evaluation

`POST /api/v1/debug/evaluations/runs`

```json
{
  "case_set_id": "phase3-default",
  "case_set_version": "0.3.0",
  "asset_package_id": "phase2-default",
  "asset_package_version": "0.2.0",
  "include_tags": ["high_risk"],
  "fail_fast": false
}
```

### 4. 生成 Candidate 并 Review

```text
POST /api/v1/debug/candidates/generations/from-evaluation/{run_id}
GET /api/v1/debug/candidates/generations/{generation_id}
POST /api/v1/debug/candidates/training-example-candidates/{candidate_id}/review
GET /api/v1/debug/candidates/{candidate_id}/reviews
```

### 5. 启动前端 Console（Phase 5-P2）

```powershell
cd console-web
npm install
npm run dev
```

浏览器打开 `http://localhost:5173`，在侧边栏 **Debug Context** 配置 Token / Actor / Roles，可演示五页治理界面与 Review 操作。

### 6. Phase 5 当前状态

Phase 5-P1 Console API 与 Phase 5-P2 前端 MVP 均已交付并冻结。归档见 `docs/Phase5冻结记录.md`；验收见 `docs/Phase5_P2人工测试结果.md`。

# 七、和普通 Agent / RAG 的区别

| 对比点 | 普通 RAG / Agent | ClinMindRuntime |
|---|---|---|
| 主控 | LLM / Agent | Java Runtime Core |
| 状态 | 多数弱状态 | RuntimeState |
| 安全 | Prompt 约束为主 | SafetyGate + DecisionBoundary |
| 知识 | 检索后拼 Prompt | EvidenceRef → EvidenceGraph |
| Agent | 自主规划和执行 | Runtime 授权下生成 Proposal |
| 输出 | 通常单一回答 | PatientOutput / ClinicianReport 分角色 |
| 追踪 | 日志为主 | RuntimeTrace + asset version + AuditLog |
| 能力证明 | Demo 效果 | EvaluationCaseSet + Scorer + EvaluationResult |
| 经验沉淀 | 常常自动记忆 | Candidate + Review Required |
| 治理 | 弱治理 | CandidateSanitizer + SourceRef + Review + AuditLog |
| 持久化 | 可选日志 | PostgreSQL-backed governance runtime |
| Console | 后补管理页 | RBAC-lite + Safe DTO + Audit Center + console-web/ 最小 MVP |

# 十、项目边界声明

当前项目是：

```text
医疗 AI Runtime Core + Asset Governance + Evaluation Loop + Candidate Governance + Persistence Governance + Minimal Console Governance 的工程原型。
```

在总设计 v2.2 中，它的长期定位是：

```text
受控医疗 AI Agent Runtime 与能力治理平台。
```

当前项目不是完整产品化系统，也不替代真实医疗流程。

它展示的是：

```text
AI 能力如何在高风险场景中被控制、评估、追踪、复盘、审核、持久化治理，并逐步进入可访问治理边界。
```
