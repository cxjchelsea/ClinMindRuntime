> 本文档用于快速说明 ClinMindRuntime 的项目定位、当前实现状态、核心架构、验证方式和展示重点。  
> 它是项目展示入口，不替代完整系统设计、技术实现总方案和各 Phase 详细设计。

---

# 一、项目一句话介绍

ClinMindRuntime 是一个面向医疗问诊场景的受控 AI Runtime 原型。

它的重点不是让大模型直接生成最终回答，而是让 AI 能力在一个可控、可评估、可追踪、可回滚的 Runtime 中运行。

```text
普通 RAG / Agent：输入 → 检索 / 工具 → LLM → 输出
ClinMindRuntime：输入 → RuntimeState → SafetyGate → EvidenceGraph → DecisionBoundary → 分角色输出 → RuntimeTrace → Evaluation
```

---

# 二、项目要解决的问题

在高风险问诊场景中，AI 系统不能只追求回答自然，还需要处理：

```text
1. 高风险信号不能被忽略。
2. 患者端不能看到内部推理和医生端信息。
3. 知识、规则、经验、模型需要有版本和来源。
4. 每次输出需要能追踪依据。
5. 能力需要通过病例集和指标评估。
6. 外部模型、RAG、MCP 等能力不能绕过 Runtime 主控。
```

# 四、核心架构链路

## 4.1 Runtime 主链路

```text
User Input
→ RuntimeService
→ EntryAssessment
→ CaseFrame
→ KnowledgeContext
→ ExperienceContext
→ SafetyGate
→ DifferentialDiagnosisBoard
→ EvidenceGraph
→ QuestionTestPolicy
→ DecisionBoundary
→ PatientOutput / ClinicianReport
→ RuntimeTrace
```

## 4.2 Asset Provider 链路

```text
Runtime Core
→ Provider Interface
→ YAML Asset Package
→ versioned asset refs
→ RuntimeTrace assets_used
```

## 4.3 Evaluation 闭环

```text
EvaluationCaseSet
→ RuntimeEvaluationRunner
→ RuntimeService.startRuntime / continueRuntime
→ RuntimeState / RuntimeTrace / API Response
→ EvaluationScorer
→ EvaluationItemResult
→ EvaluationResult
→ CapabilityProfileUpdateProposal
```

# 五、当前最值得展示的能力

## 5.1 受控 Runtime，而不是普通问答链

一次交互会进入 Runtime 实例，并维护状态、证据、安全门、输出边界和 Trace。

## 5.2 患者端 / 医生端输出隔离

患者端只返回安全表达，医生端可以看到更多内部辅助信息。

## 5.3 Asset Package 版本治理

系统会绑定 asset_package_id 和 asset_package_version，版本不匹配时进入安全处理，而不是静默继续。

## 5.4 EvaluationScorer 策略体系

当前已有评分维度：

```text
EntryAssessmentScorer
SafetyGateScorer
PatientBoundaryScorer
DdxCoverageScorer
NextActionScorer
TraceCompletenessScorer
AssetVersionTraceScorer
```

## 5.5 CapabilityProfileUpdateProposal

EvaluationResult 可以生成能力档案更新建议，但不会自动上线。

# 六、如何验证 Phase 3 Evaluation

创建评估运行：

```http
POST /api/v1/debug/evaluations/runs
```

示例请求：

```json
{
  "case_set_id": "phase3-default",
  "case_set_version": "0.3.0",
  "asset_package_id": "phase2-default",
  "asset_package_version": "0.2.0",
  "include_tags": ["high_risk"],
  "fail_fast": false
}
```

查询评估运行：

```http
GET /api/v1/debug/evaluations/runs/{run_id}
```

查询评估结果：

```http
GET /api/v1/debug/evaluations/runs/{run_id}/result
```

查询单病例结果：

```http
GET /api/v1/debug/evaluations/runs/{run_id}/items/{case_id}
```

生成能力档案建议：

```http
POST /api/v1/debug/evaluations/runs/{run_id}/capability-profile-proposal?symptom_group=chest_pain
```

详细人工验收见：

```text
docs/Phase3_人工测试API结果.md
```

# 七、和普通 Agent / RAG 的区别

| 对比点 | 普通 RAG / Agent | ClinMindRuntime |
|---|---|---|
| 主控 | LLM / Agent | Java Runtime Core |
| 状态 | 多数弱状态 | RuntimeState |
| 安全 | Prompt 约束为主 | SafetyGate + DecisionBoundary |
| 知识 | 检索后拼 Prompt | EvidenceRef → EvidenceGraph |
| 输出 | 通常单一回答 | PatientOutput / ClinicianReport 分角色 |
| 追踪 | 日志为主 | RuntimeTrace + asset version |
| 能力证明 | Demo 效果 | EvaluationCaseSet + Scorer + EvaluationResult |
| 能力升级 | 手动或隐式 | CapabilityProfileUpdateProposal |

# 十、项目边界声明

当前项目是：

```text
医疗 AI Runtime Core + Asset Governance + Evaluation Loop 的工程原型。
```

当前项目不是完整产品化系统，也不替代真实医疗流程。

它展示的是：

```text
AI 能力如何在高风险场景中被控制、评估、追踪和治理。
```
