# ClinMindRuntime 全局技术栈与架构选型

> 本文档用于定义 ClinMindRuntime 的长期技术栈、技术边界、接入阶段和替换策略。  
> 它不是要求当前一次性实现所有技术，而是约束后续 Phase 在引入前端、数据库、向量数据库、图数据库、Python AI Provider、RAG、异步任务和部署体系时不偏离整体架构。

---

# 一、文档定位

ClinMindRuntime 不是单纯的 Python AI Demo，也不是普通后端 CRUD 系统。

它的长期定位是：

```text
以 Java / Spring Boot 为 Runtime 主控，
以 Provider 机制接入 AI 能力、知识资产、评估能力和治理能力，
以 PostgreSQL 等基础设施承载资产、评估、Trace、审计和平台治理，
以 Python 作为模型能力与实验能力的 Provider / Experiment 层。
```

因此，全局技术栈必须同时满足：

```text
1. Runtime 主控稳定。
2. 医疗安全边界可控。
3. AI 能力可替换。
4. 资产、评估、经验和治理可持续演进。
5. 不因过早引入复杂基础设施而破坏 P0 主线。
```

---

# 二、总原则

```text
1. Java / Spring Boot 是 Runtime Core 主工程。
2. Python 是 AI Provider / Experiment 层，不是 Runtime 主控。
3. LLM / RAG / KG / embedding 只能作为 Provider 或资产来源，不得绕过 RuntimeState、SafetyGate、DecisionBoundary。
4. Phase 1–3 以最小闭环为主，不提前引入数据库、前端、向量库、图数据库和消息队列。
5. PostgreSQL 是长期主数据库默认选型。
6. pgvector 是小规模向量能力默认优先方案，Milvus / Qdrant 作为后续可替代方案。
7. KG-lite 优先用 PostgreSQL 表结构表达，Neo4j 后置。
8. 前端平台后置到 Phase 5，不提前干扰 Runtime / Asset / Evaluation 主线。
9. 所有技术引入都必须说明：为什么现在需要、服务哪个 Phase、是否破坏已有边界。
```

# 三、分层架构与技术职责

```text
Frontend / Console Layer
  React + TypeScript
  负责 Runtime Console、Training Center、Asset Console、Audit Center 等平台页面。

API / Runtime Core Layer
  Java 17+ / Spring Boot 3.x
  负责 Runtime API、状态流转、安全门、输出边界、Trace、Evaluation、资产治理入口。

Provider Layer
  Java Provider / Python FastAPI Provider
  负责 LLM、RAG、embedding、结构化抽取、证据检索、相似病例检索等可替换能力。

Data / Governance Layer
  PostgreSQL / Redis / pgvector / 可选 Neo4j / 对象存储
  负责会话、Trace、资产版本、评估结果、经验候选、审计和治理。

Experiment Layer
  Python notebooks / scripts / FastAPI service
  负责离线分析、模型实验、指标可视化、embedding 实验、候选经验挖掘。
```

# 五、Java 与 Python 的职责边界

## 5.1 Java 负责什么

```text
1. Runtime 主控。
2. RuntimeState 生命周期。
3. RuntimeStatus 状态流转。
4. SafetyGate 强规则兜底。
5. DecisionBoundary 输出边界。
6. Patient / Clinician 输出隔离。
7. RuntimeTrace 与审计链路。
8. Asset Package / Provider 接入边界。
9. EvaluationRunner 和结构化评估主流程。
10. CapabilityProfile proposal 的生成与治理入口。
11. API、权限、审计和后续平台治理。
```

## 5.2 Python 负责什么

```text
1. LLM 调用实验。
2. CaseFrame 抽取实验。
3. symptom normalization。
4. embedding 生成。
5. RAG 检索实验。
6. 相似病例检索。
7. RuntimeTrace 离线分析。
8. 经验候选挖掘。
9. 评估指标可视化。
10. AI Provider 服务化。
```

## 5.3 Python 不允许做什么

```text
1. 不允许成为 Runtime 主控。
2. 不允许直接修改 RuntimeState。
3. 不允许绕过 SafetyGate。
4. 不允许绕过 DecisionBoundary。
5. 不允许直接生成患者端最终诊断结论。
6. 不允许直接上线 CapabilityProfile。
7. 不允许直接写入生产资产包。
```

    services/
      llm_client.py

      trace_analysis_service.py
  pyproject.toml
```

候选接口：

```text
POST /ai/extract-case-frame
POST /ai/normalize-symptom
POST /ai/generate-embedding
POST /ai/retrieve-evidence
POST /ai/retrieve-similar-cases
POST /ai/analyze-runtime-trace
POST /ai/suggest-experience-candidates
```

返回原则：

```text
Python Provider 只能返回结构化 Draft / Candidate / EvidenceRef。
Java Runtime 决定是否采纳。
患者端最终输出仍由 Java Runtime 的 DecisionBoundary 控制。
```

推荐接入阶段：

```text
Phase 1-P0：不接 Python。
Phase 2-P0：不接 Python。
Phase 3-P0：不接 Python，先用 Java 评估 Runtime。
Phase 3-P1：可用 Python 做离线评估分析和可视化。
Phase 4-P0：可引入 Python 做经验候选挖掘、embedding、相似病例检索。
Phase 5-P0：可将 Python 能力服务化为 AI Provider。
```

# 七、数据库选型

## 7.1 P0 阶段为什么不用数据库

Phase 1–3 的重点是验证主线闭环：

```text
Runtime 是否成立。
Asset Provider 是否成立。
Evaluation 是否成立。
```

因此 P0 阶段使用：

```text
In-memory Store
YAML / JSON resources
JUnit / Postman 验收
```

这样可以避免基础设施复杂度掩盖系统主线。

## 7.2 长期主数据库：PostgreSQL

推荐 PostgreSQL 作为长期默认数据库。

用途：

```text
1. Runtime 会话。
2. RuntimeTrace。
3. AssetPackage / AssetMetadata。
4. EvaluationCaseSet。
5. EvaluationRun / EvaluationResult。
6. CapabilityProfileUpdateProposal。
7. ExperienceCandidate。
8. AuditLog。
9. User / Role / Permission。
10. JSONB 结构化扩展字段。
```

不建议长期默认 MySQL 的原因：

```text
PostgreSQL 对 JSONB、复杂查询、pgvector、审计型数据和后续扩展更适合本项目。
```

## 7.3 Redis

Redis 后置，用于：

```text
1. 分布式 Runtime session cache。
2. 短期上下文缓存。
3. 并发锁。
4. API rate limit。
5. 评估任务进度缓存。
```

推荐接入阶段：

```text
Phase 5 或 Phase 1/3 并发增强时再接入。
```

# 八、向量数据库选型

## 8.1 默认优先：pgvector

小规模阶段优先使用：

```text
PostgreSQL + pgvector
```

适用：

```text
1. 医疗证据片段 embedding。
2. 相似病例检索。
3. RuntimeTrace 相似错误检索。
4. ExperienceCandidate 检索。
```

理由：

```text
1. 减少基础设施数量。
2. 与主数据库统一。
3. 足够支撑中小规模原型。
4. 更适合当前项目的治理型数据结构。
```

## 8.2 后续可替代：Milvus / Qdrant

只有在以下条件出现时再考虑：

```text
1. 向量数量显著增大。
2. 检索 QPS 明显高。
3. 需要更复杂 ANN 索引能力。
4. pgvector 性能无法满足。
```

Phase 1–3 禁止引入独立向量数据库。

# 九、图数据库与 KG-lite 选型

## 9.1 Phase 1–3 不引入图数据库

当前 KG-lite 只保留：

```text
EvidenceAssetRef
clinical_pathway_refs
kg_lite_ref
```

不接真实 KG 引擎。

## 9.2 优先方案：PostgreSQL 表结构

后续 KG-lite 优先用：

```text
kg_nodes
kg_edges
kg_evidence_links
kg_rule_links
```

表达：

```text
症状 → 诊断
诊断 → 检查
红旗信号 → 风险动作
证据 → 诊断
经验 → 场景
```

## 9.3 后置方案：Neo4j

只有当需要复杂图遍历时才考虑：

```text
1. 多跳推理。
2. 复杂路径查询。
3. 图算法。
4. 大规模医学关系网络。
```

Phase 1–4 不默认引入 Neo4j。

# 十、前端技术选型

## 10.1 默认前端选型

推荐：

```text
React + TypeScript + Vite
```

原因：

```text
1. 轻量。
2. 适合管理台 / Console。
3. 不引入过重 SSR 复杂度。
4. 与用户现有 React / TypeScript 技能匹配。
```

## 10.2 Next.js 的位置

Next.js 可作为后续替代，但不是默认必选。

适用场景：

```text
1. 需要更完整路由体系。
2. 需要 SSR / SSG。
3. 需要更正式产品化站点。
```

ClinMindRuntime 的平台后台优先使用 Vite 即可。

## 10.3 前端接入阶段

```text
Phase 1–3：不做前端。
Phase 4：可选做简单 Evaluation / Trace 可视化页面。
Phase 5：正式做 Training Center、Runtime Console、Asset Console、Audit Center。
```

# 十一、LLM / RAG 框架边界

允许：

```text
1. Spring AI 作为 Java 侧 LLM Provider。
2. LangChain4j 作为 Java 侧 LLM / RAG Provider。
3. LangChain / LlamaIndex 作为 Python 侧实验 Provider。
4. 自定义轻量 Provider。
```

禁止：

```text
1. 禁止 LangChain / LangGraph / Spring AI / LangChain4j 成为 Runtime 主控。
2. 禁止 LLM 直接决定 SafetyGate 最终结果。
3. 禁止 LLM 直接决定 DecisionBoundary。
4. 禁止 RAG 检索结果绕过 EvidenceGraph。
5. 禁止 Provider 返回自然语言后直接作为患者端最终输出。
```

推荐原则：

```text
LLM / RAG 负责生成候选信息。
Runtime 负责验证、约束、追踪和输出边界。
```

# 十二、异步任务与消息队列

## 12.1 Phase 1–3 不引入 MQ

当前不需要：

```text
RabbitMQ
Kafka
RocketMQ
Nacos
Spring Cloud
```

## 12.2 轻量优先

后续如需异步评估或批量任务，优先：

```text
Spring Scheduler
Database Job Table
In-memory Job Store
```

## 12.3 MQ 接入条件

只有出现以下需求再引入 MQ：

```text
1. 评估任务耗时较长且需要异步队列。
2. 多服务之间事件解耦。
3. 经验候选生成与审核流程需要事件驱动。
4. 审计和回放任务吞吐明显增加。
```

选型建议：

```text
RabbitMQ：中小规模任务队列。
Kafka：高吞吐事件流和日志流。
```

# 十七、技术替换策略

任何技术替换都必须满足：

```text
1. 不破坏 Runtime 主控。
2. 不破坏患者端输出边界。
3. 不破坏 SafetyGate / DecisionBoundary。
4. 不破坏 RuntimeTrace / asset version trace。
5. 有明确收益，而不是因为技术流行。
6. 有回归测试。
7. 有文档说明替换原因。
```

示例：

```text
pgvector → Milvus：只有当向量规模和检索性能确实需要时才替换。
PostgreSQL KG-lite → Neo4j：只有复杂图遍历成为核心需求时才替换。
Vite → Next.js：只有正式产品站点需要 SSR / 更复杂路由时才替换。
In-memory Store → PostgreSQL：只有进入持久化、审计、平台化阶段才替换。
```

# 十八、最终结论

```text
ClinMindRuntime 的主线不是“用 Python 快速做 AI Demo”，
而是构建一个医疗 AI Runtime 与能力治理系统。

Java / Spring Boot 负责稳定主控、安全边界、状态、Trace、资产和评估。
Python 负责模型能力、RAG、embedding、离线实验和后续 AI Provider。
PostgreSQL 是长期治理型数据底座。
pgvector 是小规模向量检索优先方案。
KG-lite 先用 PostgreSQL 表结构表达，Neo4j 后置。
前端平台后置到 Phase 5。

全局技术路线是：先闭环，再接基础设施；先 Provider 边界，再接 AI 能力；先评估授权，再做经验进化和平台治理。
```
