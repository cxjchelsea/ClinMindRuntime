# 全项目功能点人工测试手册：ClinMindRuntime 当前实现覆盖版

> 适用范围：当前仓库已经实现的 Phase 1–10 P0/P1/P2 相关能力。  
> 测试目标：不是测试一个尚未实现的“问诊前端产品”，而是逐点测试当前项目已经实现的 Runtime、Agent、Evidence、Provider、Model Governance、Tool Governance、Evaluation、Candidate、Audit、Console 等功能。  
> 测试方式：前端页面 + Debug API + Console 观测 + 必要的自动化命令。  
> 明确排除：当前尚未实现正式问诊交互前端，因此本手册不测试“患者输入症状 → 多轮问诊 UI → 诊断页面”这一产品流程。

---

# 一、先理解：当前项目到底能测什么

当前项目不是普通医疗聊天机器人，而是：

```text
受控医疗 AI Agent Runtime 与能力治理平台
```

所以当前已经实现的功能大致分为四类：

| 类别 | 已实现内容 | 显式人工测试方式 |
|---|---|---|
| Runtime 与状态链路 | RuntimeState、RuntimeTrace、SafetyGate、DecisionBoundary、PatientOutput/ClinicianReport 边界对象 | 通过 Console Runtime Sessions / Runtime Timeline 观察；通过 API 造数据后观察 |
| 能力层 | Agent、RAG Evidence、Graph Evidence、Python Provider、Judge、Risk、Tool / MCP / Skills | 主要通过 Debug API 调用，再通过 Console 观察 Audit / Timeline / Candidate |
| 治理层 | Evaluation、Candidate、Review、Audit、ModelRegistry、PromptRegistry、DatasetVersion、ToolRegistry | 通过 Debug API + Console 页面测试 |
| 展示层 | Governance Console / Runtime Console | 通过前端页面点击测试 |

当前还没有实现的显式产品入口：

```text
患者问诊页面
医生正式诊断页面
正式医生审核工作站
真实临床知识库管理页面
真实 MCP / EHR / HIS / LIS / PACS 接入页面
```

因此，本手册会按“功能点”测试，而不是按“问诊产品页面”测试。

## 2.3 Debug Context 设置

打开：

```text
http://localhost:5173
```

左侧应出现 `Debug Context` 面板。

设置：

| 字段 | 建议输入 |
|---|---|
| API Base URL | 留空，使用 Vite 代理；或填 `http://localhost:8080` |
| X-Debug-Actor | `manual-tester` |
| X-Debug-Roles | 勾选 `SYSTEM_ADMIN` |
| X-Debug-Token | 如果后端配置 require debug token，则填写；否则留空 |

点击：

```text
测试连接
```

预期结果：

```text
连接成功
```

如果失败，检查：

```text
1. 后端是否启动。
2. API Base URL 是否正确。
3. 是否需要 X-Debug-Token。
4. 浏览器 Network 中 /api/v1/debug/console/runtime-sessions?limit=1 是否成功。
```
