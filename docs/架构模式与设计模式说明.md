# 架构模式与设计模式说明

> 本文档用于帮助理解 ClinMindRuntime 的架构模式、设计模式和代码实现模式。  
> 它不是 Phase 3-P0 的实现规格，也不是新增功能清单；它的价值在于帮助开发者理解：为什么当前代码要这样分层、为什么要有 Runtime、Provider、Asset、Evaluation、SafetyGate、DecisionBoundary、Trace，以及这些设计分别对应哪些软件工程模式。

---

# 一、为什么需要理解“模式”

软件工程里的“模式”不是为了套概念，而是为了回答三个问题：

```text
1. 为什么系统要这样拆？
2. 为什么模块之间要这样依赖？
3. 为什么代码要用这种固定写法？
```

在 ClinMindRuntime 中，模式可以分为三层：

```text
架构模式：决定系统大的边界和运行方式。
设计模式：决定模块之间如何解耦、替换、扩展和治理。
代码模式：决定具体代码如何组织、命名、返回、测试、兜底和追踪。
```

对应关系：

```text
架构模式 = 系统骨架
设计模式 = 模块协作方式
代码模式 = 代码书写习惯和工程约束
```

如果不理解这些模式，项目容易被看成“一堆 Service 串起来”。

如果理解这些模式，就会发现 ClinMindRuntime 的核心不是“医疗问答”，而是：

```text
一个受控医疗 AI Runtime，
通过架构模式控制边界，
通过设计模式解耦能力，
通过代码模式保证可追踪、可评估、可回滚。
```

# 三、架构模式一：受控 Runtime 架构

## 3.1 这是什么

ClinMindRuntime 最核心的架构模式是：

```text
Controlled Medical AI Runtime Architecture
受控医疗 AI Runtime 架构
```

它不是普通的：

```text
Controller → Service → 调 LLM → 返回答案
```

也不是普通的：

```text
User Input → RAG → Prompt → LLM → Patient Answer
```

而是：

```text
User Input
→ RuntimeService
→ RuntimeState
→ Domain Modules
→ Provider / Asset
→ SafetyGate
→ EvidenceGraph
→ DecisionBoundary
→ PatientOutput / ClinicianReport
→ RuntimeTrace
```

核心区别：

```text
LLM / RAG / Model 不是主控。
Runtime 才是主控。
```

## 3.2 为什么要这样设计

医疗场景有几个天然约束：

```text
1. 不能让模型自由决定是否输出诊断。
2. 不能让患者端看到医生端的推理细节。
3. 不能把高风险病例误当成普通咨询。
4. 不能让知识库检索结果直接变成医疗建议。
5. 不能让能力边界靠“我觉得模型可以”来决定。
```

所以系统必须有一个主控层：

```text
RuntimeState：知道当前病例状态。
SafetyGate：知道是否触发安全风险。
DecisionBoundary：知道当前能不能说、能说到什么程度。
RuntimeTrace：记录为什么这么判断。
EvaluationResult：证明这个能力是否可靠。
```

## 3.3 在代码中的体现

```text
RuntimeService 是主流程入口。
RuntimeState 是状态载体。
SafetyGateService 是安全门。
DecisionBoundaryService 是输出边界。
PatientOutputService 只负责生成患者安全表达。
ClinicianReportService 生成医生端信息。
RuntimeTrace 记录模块执行和资产使用。
```

## 3.4 面试表达

可以这样说：

```text
这个项目的核心不是让大模型直接回答医学问题，而是把 AI 能力放进一个受控 Runtime 中运行。
Runtime 负责状态、安全、证据、输出边界和 Trace，模型、RAG、MCP 都只能作为 Provider 返回候选结果，不能直接决定患者端输出。
```

# 四、架构模式二：分层架构

## 4.1 分层结构

ClinMindRuntime 使用的是分层架构：

```text
API Layer
→ Application / Runtime Layer
→ Domain Module Layer
→ Provider / Asset Layer
→ Storage / Repository Layer
→ Integration Layer
```

每一层负责不同问题。

## 4.2 API Layer

负责：

```text
接收请求
参数校验
调用应用服务
返回 DTO
错误响应映射
```

不负责：

```text
不做医学判断。
不直接调用 SafetyGate。
不直接读 Provider。
不直接返回 RuntimeState 给患者端。
```

## 4.3 Application / Runtime Layer

负责：

```text
编排一次 Runtime 执行。
控制模块调用顺序。
更新 RuntimeState。
触发 Trace。
调用 EvaluationRunner。
```

典型类：

```text
RuntimeService
EvaluationRunService
CapabilityProfileProposalService
```

## 4.4 Domain Module Layer

负责领域判断：

```text
EntryAssessment
CaseFrame
SafetyGate
DifferentialDiagnosisBoard
EvidenceGraph
QuestionTestPolicy
DecisionBoundary
Output
```

这里才是业务核心。

## 4.5 Provider / Asset Layer

负责把外部能力或资产变成 Runtime 可用对象：

```text
MedicalKnowledgeProvider
ClinicalExperienceProvider
CapabilityProfileProvider
EvidenceAssetProvider
```

未来也包括：

```text
RagProvider
GraphRagProvider
PythonModelProvider
McpAdapter
```

## 4.6 Storage / Repository Layer

负责数据读取和保存：

```text
RuntimeStore
AssetPackageRepository
EvaluationCaseRepository
EvaluationRunStore
```

当前 Phase 1–3 用 in-memory 和 YAML；Phase 4/5 再进入 PostgreSQL、Redis、pgvector。

## 4.7 Integration Layer

后置层，用于外部系统：

```text
PythonAiProviderClient
ExternalLlmClient
McpClient
RagProviderClient
EmbeddingClient
```

它不能反向控制 Runtime。

## 4.8 分层架构的价值

```text
Controller 不会越来越胖。
RuntimeService 只负责编排，不直接写死所有知识。
领域模块可以单独测试。
Provider 可以替换。
存储可以从 YAML 切到 PostgreSQL。
未来接 Python / RAG / MCP 时不会破坏 Runtime Core。
```

# 五、架构模式三：Ports and Adapters / 六边形架构思想

## 5.1 本项目里的 Port 和 Adapter

在 ClinMindRuntime 中：

```text
Runtime Core = 应用核心
Provider Interface = Port
YAML Provider / RAG Provider / Python Provider / MCP Adapter = Adapter
```

例如：

```text
Port：EvidenceAssetProvider
Adapter 1：YamlEvidenceAssetProvider
Adapter 2：RagEvidenceProvider
Adapter 3：GraphRagEvidenceProvider
Adapter 4：McpEvidenceAdapter
```

## 5.2 为什么不是直接调用外部能力

错误做法：

```text
RuntimeService
→ 直接读 YAML
→ 直接调 RAG
→ 直接调 LLM
→ 直接拼回答
```

问题：

```text
Runtime 和外部能力耦合。
测试困难。
替换困难。
错误难以隔离。
无法统一追踪版本。
```

正确做法：

```text
RuntimeService
→ EvidenceAssetProvider interface
→ provider implementation
→ YAML / RAG / Python / MCP
```

这样 Runtime 只依赖抽象。

## 5.3 这个模式解决什么问题

```text
1. 当前可以用 YAML mock。
2. 后续可以换 PostgreSQL。
3. 再后续可以接 RAG。
4. 还可以接 MCP。
5. Runtime 主流程不用重写。
```

## 5.4 面试表达

```text
我在 Phase 2 没有直接接真实 RAG，而是先抽象 Provider 接口。这样 Runtime Core 不依赖具体知识来源，当前可以用 YAML 资产包，后续可以平滑切换到 PostgreSQL、RAG、GraphRAG 或 MCP Adapter。这本质上是 Ports and Adapters 的思想。
```

# 六、架构模式四：状态机 / Runtime Instance

## 6.1 一次问诊不是一次普通请求

普通 Web 系统可能是：

```text
Request → Response
```

但问诊是连续过程：

```text
第 1 轮：用户说胸闷
第 2 轮：补充出汗
第 3 轮：补充活动后加重
第 4 轮：系统建议立即就医
```

所以需要 Runtime 实例：

```text
runtime_id
RuntimeState
RuntimeStatus
RuntimeTrace
```

## 6.2 RuntimeState 的意义

RuntimeState 不是简单 session。

它保存的是：

```text
当前工作模式
症状群
病例结构化信息
候选诊断状态
证据图
安全门结果
输出边界
使用过的资产
```

## 6.3 RuntimeStatus 的意义

状态控制系统是否继续执行：

```text
CLINICAL_MODE
WELLNESS_MODE
EMERGENCY_HINT
ERROR_SAFE_HALTED
COMPLETED
```

其中 `ERROR_SAFE_HALTED` 很重要，它表示：

```text
系统不确定或出错时，宁愿停止危险输出，也不继续冒险生成答案。
```

## 6.4 与普通 Workflow 的区别

普通 Workflow 关心：

```text
步骤 A 完成后走步骤 B。
```

Runtime 更关心：

```text
当前状态是什么？
哪些信息已经知道？
哪些信息仍然缺失？
有没有高危信号？
当前允许输出什么？
```

所以本项目不是简单 workflow，而是：

```text
Stateful Clinical Runtime
```

# 七、架构模式五：Stateful Pipeline / Orchestrator

## 7.1 Pipeline 链路

当前 Runtime 主链路是：

```text
EntryAssessment
→ CaseFrame
→ KnowledgeContext
→ ExperienceContext
→ SafetyGate
→ DifferentialDiagnosisBoard
→ EvidenceGraph
→ QuestionTestPolicy
→ DecisionBoundary
→ Output
→ Trace
```

它像 Pipeline，但不是无状态 Pipeline。

每一步都读取并更新 RuntimeState。

## 7.2 Orchestrator 的作用

RuntimeService 是 Orchestrator。

它决定：

```text
调用哪些模块。
模块顺序是什么。
失败时如何处理。
是否进入 fail-safe。
是否允许继续输出。
```

## 7.3 为什么不用每个模块自己决定下一步

如果每个模块都自己决定下一步，就会变成：

```text
EntryAssessment 调 SafetyGate
SafetyGate 调 EvidenceGraph
EvidenceGraph 调 Output
Output 再调 DecisionBoundary
```

这会导致：

```text
依赖混乱。
流程不可控。
测试困难。
安全边界可能被绕过。
```

所以必须由 RuntimeService 统一编排。

# 八、架构模式六：Evaluation-driven Architecture

## 8.1 能力不是声明出来的

医疗 AI 项目不能说：

```text
我支持胸痛问诊。
```

而应该说：

```text
我在 chest_pain case set 上跑过 Evaluation。
Safety pass rate 是多少。
Patient boundary violation 是多少。
Trace completeness 是多少。
Asset version trace 是否完整。
是否允许提升 CapabilityProfile。
```

## 8.2 Phase 3 的闭环

```text
EvaluationCaseSet
→ EvaluationRunner
→ RuntimeService
→ RuntimeCaseExecution
→ Scorers
→ EvaluationItemResult
→ EvaluationResult
→ CapabilityProfileUpdateProposal
```

## 8.3 这是什么架构思想

这是：

```text
Evaluation-driven Architecture
Capability Governance Architecture
```

即能力必须通过评估结果来授权。

## 8.4 它和模型训练的关系

Phase 3 的“训练”不是训练大模型，而是：

```text
用标准病例集训练 / 校准 Runtime 的能力边界。
```

后续模型训练也必须被 Evaluation 驱动：

```text
TrainingDatasetVersion
→ ModelProviderVersion
→ EvaluationResult
→ CapabilityProfileUpdateProposal
→ Review
→ Runtime 可用
```

# 九、设计模式一：Strategy 策略模式

## 9.1 在 Provider 中的体现

Provider 是策略模式最明显的地方。

接口：

```text
EvidenceAssetProvider
```

不同策略：

```text
YamlEvidenceAssetProvider
RagEvidenceProvider
GraphRagEvidenceProvider
McpEvidenceAdapter
```

Runtime 只依赖接口，不关心具体策略。

## 9.2 在 Scorer 中的体现

接口：

```text
EvaluationScorer
```

不同评分策略：

```text
SafetyGateScorer
PatientBoundaryScorer
DdxCoverageScorer
NextActionScorer
TraceCompletenessScorer
AssetVersionTraceScorer
```

每个 Scorer 负责一个维度。

## 9.3 策略模式的价值

```text
新增能力时不改主流程。
替换算法时不改调用方。
可以单独测试每个策略。
可以按配置选择策略。
```

# 十、设计模式二：Adapter 适配器模式

## 10.1 什么时候需要 Adapter

只要外部系统和 Runtime 内部结构不一致，就需要 Adapter。

例如：

```text
Python FastAPI 返回 JSON。
MCP Server 返回 tool result。
RAG 服务返回 chunks。
LLM 返回自然语言。
```

但 Runtime 需要的是：

```text
EvidenceRef
CaseFrameDraft
RiskSignalCandidate
MetricResultDraft
```

Adapter 负责转换。

## 10.2 本项目里的 Adapter

```text
PythonAiProviderClient
RagProviderClient
GraphRagProvider
McpAdapter
ExternalLlmClient
```

这些都是后置 Adapter。

## 10.3 Adapter 的边界

Adapter 可以做：

```text
协议转换。
字段映射。
timeout / retry / fallback。
provider version 记录。
```

Adapter 不可以做：

```text
直接修改 RuntimeState。
直接生成 PatientOutput。
直接决定 SafetyGate。
直接上线 CapabilityProfile。
```

# 十一、设计模式三：Repository 模式

## 11.1 Repository 解决什么

Repository 把数据读取方式和业务逻辑隔离。

业务模块不应该知道数据来自：

```text
YAML
PostgreSQL
文件系统
远程 API
```

它只应该依赖 Repository 接口。

## 11.2 本项目中的 Repository

```text
RuntimeStore
AssetPackageRepository
YamlAssetPackageRepository
EvaluationCaseRepository
YamlEvaluationCaseRepository
EvaluationRunStore
```

## 11.3 为什么重要

当前是：

```text
YAML resources
in-memory store
```

未来可能是：

```text
PostgreSQL
Redis
pgvector
```

如果没有 Repository，未来切换存储会影响业务模块。

# 十二、设计模式四：Facade 门面模式

## 12.1 RuntimeService 是 Facade

Controller 不需要知道 Runtime 内部有多少模块。

它只需要调用：

```text
RuntimeService.startRuntime()
RuntimeService.continueRuntime()
RuntimeService.getRuntime()
```

RuntimeService 内部再协调：

```text
EntryAssessment
CaseFrame
KnowledgeContext
SafetyGate
DDx
EvidenceGraph
DecisionBoundary
Output
Trace
```

## 12.2 门面模式的价值

```text
降低 Controller 复杂度。
隐藏内部模块细节。
统一 Runtime 执行入口。
防止外部绕过主流程。
```

# 十三、设计模式五：Policy 模式

## 13.1 什么是 Policy

Policy 是把“规则判断”抽出来。

它回答的是：

```text
该不该做某事？
允许做到什么程度？
满足什么条件才能升级？
失败时怎么处理？
```

## 13.2 本项目中的 Policy

```text
QuestionTestPolicy
FailurePolicy
DecisionBoundary
CapabilityEvaluationPolicy
```

## 13.3 举例

```text
QuestionTestPolicy：当前应该追问，还是建议检查？
FailurePolicy：出错时继续、降级，还是 safe halt？
DecisionBoundary：患者端能否看到某类信息？
CapabilityEvaluationPolicy：EvaluationResult 是否允许生成升级建议？
```

## 13.4 为什么不用 if else 写在主流程里

如果所有规则都写在 RuntimeService 里：

```text
RuntimeService 会越来越大。
规则无法复用。
测试困难。
后续改边界容易影响主流程。
```

Policy 抽离后，每个规则可以单独测试和替换。

# 十四、设计模式六：Chain of Responsibility / Pipeline 思想

## 14.1 本项目不是纯责任链

经典责任链是：

```text
请求沿着链传递，某个 handler 处理后决定是否继续。
```

ClinMindRuntime 不是完全经典责任链，因为 RuntimeService 仍然统一编排。

但安全链路里有责任链思想：

```text
EntryAssessment 发现非临床输入 → 不进入临床 pipeline
SafetyGate 发现高危 → 收紧输出
DecisionBoundary 不允许输出 → PatientOutput 降级
FailurePolicy 发现错误 → ERROR_SAFE_HALTED
```

## 14.2 为什么这很重要

医疗场景中，任何一个安全模块发现风险，都不能被后面的生成模块冲掉。

所以安全信息要沿链路传递。

# 十五、设计模式七：Template Method 思想

## 15.1 EvaluationRunner 的固定流程

Phase 3 的 EvaluationRunner 未来会有固定流程：

```text
加载病例集
创建 EvaluationRun
逐个执行 case
调用 RuntimeService
收集 RuntimeCaseExecution
执行 Scorers
聚合 EvaluationResult
生成 CapabilityProfileUpdateProposal
```

流程固定，但具体 Scorer 可以扩展。

这就是 Template Method 思想。

## 15.2 为什么有价值

```text
Evaluation 主流程稳定。
新增评分维度时不改 Runner 主体。
每个 Scorer 单独实现。
```

# 十六、设计模式八：AOP / Decorator 思想

## 16.1 RuntimeTrace 为什么适合 AOP

Trace 是横切关注点。

它不属于某个单一业务模块，而是每个关键模块都要记录。

如果手写：

```text
entryAssessment 开始记录
entryAssessment 结束记录
safetyGate 开始记录
safetyGate 结束记录
...
```

业务代码会被污染。

## 16.2 AOP 的价值

```text
业务模块专心做业务判断。
TraceAspect 统一记录执行过程。
减少重复代码。
保证 Trace 结构统一。
```

这接近 Decorator 思想：

```text
不改原对象核心逻辑，给它包一层附加行为。
```

# 十七、设计模式九：DTO / Mapper / Output Boundary

## 17.1 为什么不能直接返回 RuntimeState

RuntimeState 内部可能包含：

```text
候选诊断
must_not_miss
EvidenceGraph
资产版本
医生端报告
安全门内部判断
```

这些不能全部给患者端。

## 17.2 正确方式

```text
RuntimeState
→ PatientResponseMapper
→ PatientRuntimeResponse
```

医生端则是：

```text
RuntimeState
→ ClinicianResponseMapper
→ ClinicianRuntimeResponse
```

debug 端：

```text
RuntimeState
→ DebugRuntimeResponse
```

## 17.3 这个模式的核心

```text
内部模型和外部响应必须隔离。
不同角色使用不同 DTO。
Mapper 是输出边界的一部分。
```

# 十八、代码模式一：统一包结构

推荐包结构：

```text
api/
runtime/
state/
trace/
entry/
caseframe/
knowledge/
experience/
safety/
reasoning/
boundary/
output/
asset/
provider/
evaluation/
training/
integration/
common/
```

这个包结构表达了架构意图：

```text
runtime = 主控

```

代码包不是随便分的，而是在表达架构边界。

# 十九、代码模式二：Result Object

## 19.1 为什么不用 boolean / string

如果 SafetyGate 只返回 boolean：

```text
true / false
```

就不知道：

```text
触发了什么风险？
证据是什么？
等级是什么？
是否应该 halt？
是否应该给患者看？
```

## 19.2 应该返回结构化 Result

例如：

```text
SafetyGateResult
- triggered
- risk_level
- triggered_rules
- evidence_refs
- recommended_action
- fail_safe_required
```

其他模块也类似：

```text
EntryAssessmentResult
CaseFrameResult
DecisionBoundaryResult
MetricResult
EvaluationItemResult
```

## 19.3 Result Object 的价值

```text
可测试。
可追踪。
可扩展。
可序列化。
可进入 Evaluation。
```

# 二十、代码模式三：Enum 状态

## 20.1 为什么状态不能用字符串

字符串容易出现：

```text
clinical_mode
CLINICAL_MODE
clinicalMode
clinic_mode
```

会导致判断错误。

## 20.2 应该使用 Enum

```text
RuntimeStatus
WorkMode
CaseSeverity
MetricSeverity
SafetyViolationType
EvaluationRunStatus
AssetStatus
ReviewStatus
```

## 20.3 Enum 的价值

```text
限制状态范围。
提高可读性。
便于 switch / policy 判断。
便于测试覆盖。
```

# 二十一、代码模式四：Fail-safe / Fail-closed

## 21.1 医疗系统不能 fail-open

错误方式：

```text
规则加载失败 → 忽略规则 → 继续正常输出
```

这叫 fail-open。

医疗场景更应该：

```text
规则加载失败 → fail-safe → 不输出危险内容
```

## 21.2 本项目中的 fail-safe

```text
坏资产包加载失败 → fail-closed
asset_context.version mismatch → ERROR_SAFE_HALTED
DecisionBoundary 异常 → 安全降级
Provider timeout → fallback / safe halt
```

## 21.3 价值

```text
不确定时不冒险。
错误不能扩大输出权限。
安全模块失败不能静默忽略。
```

# 二十二、代码模式五：Version Binding

## 22.1 为什么必须绑定版本

如果系统输出了某个建议，必须知道：

```text
用了哪个资产包？
用了哪个规则版本？
用了哪个 Provider 版本？
用了哪个模型版本？
在哪个 EvaluationRun 中验证过？
```

否则无法复盘。

## 22.2 当前版本字段

```text
asset_package_id
asset_package_version
asset_id
asset_version
provider_version
case_set_version
model_provider_version
training_dataset_version
```

## 22.3 版本绑定的价值

```text
可追踪。
可评估。
可回滚。
可复现。
可审计。
```

# 二十三、代码模式六：Regression Test

## 23.1 每个 bug 都应该变成回归测试

例如：

```text
患者端泄露诊断字段 → PatientOutputAssetIsolationTest
资产版本不匹配 → RuntimeAssetVersionMismatchTest
坏资产包加载失败 → BrokenAssetPackageFailSafeTest
```

## 23.2 为什么重要

医疗 AI Runtime 的质量不是靠一次测试，而是靠长期回归。

每修一个问题，都要让这个问题以后不能再出现。

# 二十四、模式之间的关系

这些模式不是孤立的。

它们互相支撑：

```text
分层架构
保证 Controller、Runtime、Domain、Provider 不混在一起。

Ports and Adapters
保证外部知识、模型、RAG、MCP 可替换。

Runtime 状态机
保证一次问诊有状态、有边界、有上下文。

Pipeline / Orchestrator
保证执行顺序统一，不让模块乱调。

Policy
保证安全规则和能力授权规则可测试、可替换。

DTO / Mapper
保证患者端和医生端输出隔离。

AOP Trace
保证执行过程可追踪。

Evaluation-driven
保证能力必须被病例集和指标验证。

Fail-safe
保证出错时不扩大风险。

Version Binding
保证输出可以复盘、回滚和审计。
```

一句话总结：

```text
架构模式控制系统边界，
设计模式控制模块关系，
代码模式控制实现一致性。
```

# 二十五、和普通项目的区别

## 25.1 普通 RAG 项目

```text
User Question
→ Retrieve Docs

→ LLM Answer
```

特点：

```text
快。
简单。
能演示。

```

```text
User Input
→ RuntimeState

→ DecisionBoundary
→ Role-specific Output
→ RuntimeTrace
→ Evaluation
```

特点：

```text
复杂。
慢一些。

```

## 25.3 普通 Agent 项目

```text
Agent decides next step
→ Tool use
→ LLM reasoning
→ final answer
```

```text

SafetyGate / DecisionBoundary enforce boundary
Evaluation decides capability level
```

核心区别：

```text
Agent 自由规划。
ClinMindRuntime 受控执行。
```

# 二十六、面试讲法

可以这样讲：

```text
这个项目不是普通医疗问答，也不是简单 RAG。我把它设计成一个受控医疗 AI Runtime。

在架构层，我采用分层架构和 Ports and Adapters 思想。Java Runtime Core 负责状态、安全、证据、输出边界和 Trace；知识库、RAG、模型、MCP 都只能通过 Provider 接口作为外部 Adapter 接入。

在执行层，我把一次问诊建模成 Runtime 实例，而不是普通 request-response。RuntimeState 记录病例状态，SafetyGate 处理高危信号，EvidenceGraph 组织证据关系，DecisionBoundary 控制患者端和医生端分别能看到什么。

在设计模式上，我使用 Strategy 组织 Provider 和 Scorer，Repository 隔离资产读取，Adapter 接外部 AI 服务，Policy 封装安全和能力授权规则，Facade 统一 RuntimeService 入口，AOP 记录 RuntimeTrace。

在代码层，我用统一包结构、DTO/Mapper、Result Object、Enum 状态、fail-safe、asset version binding 和 regression tests 来保证系统可维护、可追踪、可评估、可回滚。

所以这个项目的重点不是让模型一次性回答得多聪明，而是让 AI 能力在医疗场景里可控地运行、可被评估、可被治理。
```
