# 一、为什么需要理解“模式”

软件工程里的“模式”不是为了套概念，而是为了回答三个问题：

```text
1. 为什么系统要这样拆？
2. 为什么模块之间要这样依赖？
3. 为什么代码要用这种固定写法？
```

在 ClinMindRuntime 中，模式可以分为三层：

```text
架构模式：决定系统大的边界和运行方式。
设计模式：决定模块之间如何解耦、替换、扩展和治理。
代码模式：决定具体代码如何组织、命名、返回、测试、兜底和追踪。
```

如果不理解这些模式，项目容易被看成“一堆 Service 串起来”。

如果理解这些模式，就会发现 ClinMindRuntime 的核心是：

```text
一个受控医疗 AI Agent Runtime 与能力治理平台，
通过架构模式控制边界，
通过设计模式解耦能力，
通过代码模式保证可追踪、可评估、可回滚、可治理。
```

# 三、核心架构模式

## 3.1 分层架构

对应总设计五层：

```text
平台治理层
共享能力资产层
Runtime 执行层
Provider / Agent / Tool 能力层
Storage / Integration 层
```

原则：

```text
上层管理和使用下层能力。
能力层不能反向控制 Runtime。
Storage / Integration 不能承载医疗决策逻辑。
Console 不能绕过 Safe DTO 和 AccessPolicy。
```

## 3.2 Ports and Adapters / 六边形架构思想

对应对象：

```text
Provider Interface
Store Interface
Repository Interface
EvidenceProvider
ModelProvider
ToolAdapter
McpAdapter
SkillProvider
```

作用：

```text
Runtime Core 依赖接口，不依赖具体外部能力。
具体实现可以从 YAML、in-memory、PostgreSQL、Python Provider、MCP Server 替换。
```

## 3.3 Orchestrator / Runtime 主控

对应对象：

```text
RuntimeService
Capability Orchestration
EvaluationRunner
CandidateGenerationService
ConsoleQueryService
```

原则：

```text
Orchestrator 负责串联流程，但不能把所有业务规则塞进一个大方法。
关键判断应交给 Policy、Validator、Boundary、Scorer 等对象。
```

## 3.4 State Machine / Stateful Pipeline

对应对象：

```text
RuntimeState
RuntimeStatus
CaseFrame
SafetyState
DecisionBoundaryResult
CandidateReviewStatus
EvaluationRunStatus
```

作用：

```text
让一次问诊不是无状态问答，而是可继续、可追踪、可复盘的状态流转。
```

## 3.5 Evaluation-driven Architecture

对应对象：

```text
EvaluationCaseSet
RuntimeEvaluationRunner
EvaluationScorer
EvaluationResult
CapabilityProfileUpdateProposal
```

原则：

```text
能力不是口头声明，而要通过病例集、指标和评估结果证明。
CapabilityProfile 不能因为模型看起来好就自动扩大。
```

## 3.6 Governance-driven Architecture

对应对象：

```text
ExperienceCandidate
TrainingExampleCandidate
CandidateReviewRecord
AuditLog
CapabilityProfileUpdateProposal
AssetPackageVersion
ModelProviderVersion
```

原则：

```text
候选不是生效。
Review 不是自动上线。
Evaluation 不是自动授权。
发布必须可审计、可回滚。
```

# 四、核心设计模式

| 模式 | 项目中的体现 | 作用 |
|---|---|---|
| Facade / Orchestrator | RuntimeService、ConsoleQueryService | 对外提供统一入口，内部编排多个模块 |
| Strategy | EvaluationScorer、QuestionPolicy、CandidateGenerationPolicy | 支持不同评分和策略替换 |
| Adapter | JdbcStore、YamlRepository、McpAdapter、PythonProviderClient | 适配不同外部实现 |
| Repository / Store | RuntimeStore、EvaluationRunStore、CandidateStore、AuditLogStore | 隔离存储实现 |
| Policy | SafetyPolicy、AccessPolicy、AgentPolicy、ToolAccessPolicy | 控制是否允许执行或输出 |
| Validator | SourceRefValidator、AgentProposalValidator、RuntimeValidation | 校验输入、来源、Proposal、外部能力结果 |
| Factory | CandidateSourceRefFactory、TraceFactory | 统一创建带约束的对象 |
| DTO Mapper | SafeConsoleDtoMapper、PatientOutputMapper | 分角色输出和敏感字段过滤 |
| Registry | AgentRegistry、ModelRegistry、ProviderRegistry | 管理可用能力和版本 |
| Template Method | EvaluationRunner 流程、Runtime pipeline | 固定主流程，允许局部扩展 |
| Fail-safe / Fail-closed | SafetyGate、asset loading failure、provider timeout | 高风险失败时不静默放行 |
| Audit Trail | AuditLogService | 记录治理动作和敏感访问 |

# 六、Phase 6-P0 需要新增的模式

Phase 6-P0 进入受控 Agent 执行层后，应新增：

```text
AgentRuntime：Controlled Execution Runtime
AgentRegistry：Registry
AgentPolicy：Policy
AgentContext：Execution Context
AgentExecutionRequest / Result：Command / Result Object
AgentProposal：Proposal Object
AgentProposalValidator：Validator
AgentTrace：Trace / Observability
AgentEvaluationHook：Evaluation Extension Point
Capability Orchestration：Planner / Orchestrator
Runtime Validation：Validation Boundary
```

核心关系：

```text
RuntimeState
→ Capability Orchestration
→ AgentPolicy
→ AgentRuntime
→ AgentProposal
→ AgentProposalValidator
→ Runtime Commit / Partial Commit / Reject / Degrade
→ RuntimeTrace / AuditLog / Evaluation
```

# 七、禁止的反模式

```text
1. God Service：把所有逻辑塞进 RuntimeService。
2. Prompt-as-Architecture：用 Prompt 代替 SafetyGate / DecisionBoundary。
3. LLM-as-Controller：让 LLM 或 Agent 决定 RuntimeState。
4. RAG-as-Answer：RAG 检索后直接回答患者。
5. Database-as-Business-Logic：用数据库触发器承担医疗决策。
6. Console-bypass-Governance：前端直接访问 raw snapshot 或绕过 Safe DTO。
7. Review-means-Published：把 review_status=APPROVED 等同于正式上线。
8. Evaluation-means-Auto-Authorization：评估结果自动扩大 CapabilityProfile。
9. Provider-reverse-dependency：Provider 反向调用 RuntimeService 或修改 RuntimeState。
10. Agent-autonomy-in-medical-core：在医疗主链路中引入自由自治式 Agent。
```

# 八、代码实现约束

```text
1. Service 只编排，不吞掉所有规则。
2. Provider 只返回结构化候选，不直接输出 PatientOutput。
3. Validator 必须显式返回拒绝原因。
4. Policy 必须可测试。
5. Store / Repository 必须保留接口和至少一种可测试实现。
6. DTO Mapper 必须明确过滤敏感字段。
7. 所有高风险失败必须 fail-safe 或 fail-closed。
8. 所有新能力必须进入 Trace / Audit / Evaluation。
9. 所有后续 AI 能力必须先进入 Phase 实现规格，再进入代码。
```

# 九、最终结论

ClinMindRuntime 的架构价值不在于用了多少框架，而在于：

```text
Runtime 有主控权，
能力有边界，
输出有角色隔离，
结果有评估，
候选有治理，
数据有持久化，
操作有审计，
新能力有 Phase 入口。
```

Phase 6 之后引入 Agent、RAG、Model、Tool / MCP / Skills 时，仍然要服从这些模式，而不能反过来让框架替代系统架构。
