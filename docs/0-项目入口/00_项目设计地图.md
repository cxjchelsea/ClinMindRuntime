# 二、当前文档整理原则

## 2.1 当前只做逻辑归档

当前不进行大规模物理搬迁。

原因：

```text
1. 现有 README、docs/README、AI_IMPLEMENTATION_SKILL、冻结记录和 Phase 文档中已有大量旧路径引用。
2. 直接移动文件会导致历史链接失效。
3. Phase 1–5 已冻结，历史文档路径应尽量保持稳定。
4. 当前重点是让文档在实现链路中“用得上”，而不是先重排目录结构。
```

因此当前采用：

```text
物理上：大部分文件仍保留在 docs/ 根目录。
逻辑上：通过本文件和 docs/README.md 将文档分为项目入口、总设计、专项设计、Phase 实现规格、冻结归档等组。
```

## 2.2 未来何时做物理归档

只有在满足以下条件时，才建议物理移动文件：

```text
1. Phase 6-P0 设计文档已经建立。
2. 需要新增大量 Phase6 / Phase7 / Phase8 文档，docs 根目录明显拥挤。
3. 愿意同步更新所有 README、冻结记录、AI_IMPLEMENTATION_SKILL 和文档内引用。
4. 能为旧路径保留迁移提示 stub，避免阅读者迷路。
```

未来如果物理归档，推荐目录结构为：

```text
docs/
  00_项目入口/
  01_总设计/
  02_专项设计/
  03_Phase实现规格/
  04_Phase冻结归档/
  05_测试验收/
```

但在当前阶段，此结构只作为逻辑分组，不作为真实目录要求。

# 三、文档分层

ClinMindRuntime 文档按五层管理。

```text
第 1 层：总设计
  定义系统是什么、由哪些能力域组成、主链路如何运行。

第 2 层：项目设计地图 / 文档导航
  说明每份文档的归属、状态、用途和实现关系。

第 3 层：专项设计文档
  展开某个能力域或工程主题，例如 Agent、RAG、模型训练、数据安全、测试、部署等。

第 4 层：Phase 实现规格与任务清单
  说明某个阶段具体实现什么、不实现什么、如何验收。

第 5 层：实现约束 / AI 编码入口 / 冻结记录
  约束 AI 编码工具、记录当前阶段状态、保存阶段完成后的冻结事实。
```

推荐阅读顺序：

```text
README.md
↓
docs/README.md
↓
docs/00_项目设计地图.md
↓
docs/ClinMindRuntime完整系统设计.md
↓
相关专项设计文档
↓
当前 Phase 实现规格
↓
当前 Phase 开发任务清单
↓
docs/AI_IMPLEMENTATION_SKILL.md
```

# 九、文档状态标记规范

后续每个专项文档开头应包含“文档定位块”。

建议格式：

```text
> 上位总设计：docs/ClinMindRuntime完整系统设计.md  
> 文档地图：docs/00_项目设计地图.md  
> 对应能力域 / 架构层：xxx  
> 当前状态：长期规划 / 子系统设计 / 当前 Phase 实现依据 / 已实现冻结  
> 当前实现：已实现 xxx，未实现 xxx  
> 对应 Phase：Phase x  
> 实现入口：进入对应 Phase 后，以 Phase*_实现规格.md 和 Phase*_开发任务清单.md 为直接开发依据。
```

这样可以避免专项文档漂在总设计之外，也避免 AI 编码工具误把长期规划当作当前任务。
