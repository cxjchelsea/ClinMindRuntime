> 上位总设计：`docs/ClinMindRuntime完整系统设计.md`

> 本项目是个人工程与架构展示项目，不宣称临床合规落地；但作为医疗 AI Runtime，必须从设计上体现数据安全意识和边界治理能力。

# 一、文档定位

本文档回答：

```text
哪些数据是敏感数据？
患者端、医生端、debug/internal API 如何隔离？
RuntimeTrace 中哪些字段需要脱敏？
EvaluationCaseSet 是否能使用真实病例？
训练数据如何匿名化？
日志、导出、审计、回放如何控制？
哪些能力必须到 Phase 5 才做完整权限和合规治理？
```

当前 Phase 1–3 不实现完整合规系统，但必须保证：

```text
患者端输出安全
debug API 路径隔离
敏感数据不随意泄露
未来权限、审计和脱敏有明确接入位置
```

---

# 二、核心原则

```text
1. 最小必要原则：只收集当前 Runtime 需要的数据。
2. 分角色输出原则：患者端、医生端、后台端看到的数据不同。
3. Debug 隔离原则：debug/internal API 不得面向 patient-facing client。
4. 可追踪原则：关键数据访问、资产使用、输出边界判断必须可审计。
5. 可脱敏原则：RuntimeTrace、Evaluation、训练数据候选必须支持脱敏。
6. 不使用真实敏感病例原则：个人项目默认使用合成病例、脱敏病例或公开样例。
7. 模型训练不得直接使用未审核、未脱敏的 RuntimeTrace。
```

# 三、数据分类

## 3.1 患者输入数据

```text
主诉
症状描述
年龄 / 性别
病程
既往史
用药史
检查结果
生活习惯
对话内容
```

风险等级：中高。

处理原则：

```text
只进入 RuntimeState。
日志中不得直接打印完整原文。
用于训练前必须脱敏和审核。
```

## 3.2 Runtime 状态数据

```text
RuntimeState
CaseFrame
RuntimeStatus
DDx Board
EvidenceGraph
SafetyGate result
DecisionBoundary result
```

风险等级：高。

处理原则：

```text
患者端不可见完整 DDx / EvidenceGraph。
医生端可见但需标注能力边界。
debug API 可见但必须内部分离。
```

## 3.3 RuntimeTrace

RuntimeTrace 包含：

```text
输入摘要
模块执行记录
知识资产引用
Provider 输出
安全门判断
输出边界判断
错误信息
```

风险等级：高。

处理原则：

```text
Trace 可用于评估和复盘。
Trace 不应直接作为训练数据。
Trace 入库前应支持字段级脱敏。
Trace 导出必须记录审计。
```

## 3.4 医生反馈与随访结局

```text
Doctor Feedback
Final Diagnosis
Correction Reason
Follow-up Outcome
Treatment Outcome
```

风险等级：高。

处理原则：

```text
必须脱敏后才可进入 ExperienceCandidate 或 TrainingExampleCandidate。
医生身份和患者身份应分离存储。
```

## 3.5 Evaluation 数据

```text
EvaluationCaseSet
ExpectedOutcome
EvaluationItemResult
SafetyViolation
RegressionFinding
```

风险等级：中。

处理原则：

```text
默认使用合成病例或公开脱敏样例。
不直接复制真实患者资料。
```

## 3.6 训练数据候选

```text
TrainingExampleCandidate
TrainingDatasetVersion
PreferencePair
ModelProviderMetadata
```

风险等级：高。

处理原则：

```text
训练样本必须来源可追踪。
进入训练集前必须脱敏、审核、版本化。
模型输出必须继续受 Runtime 约束。
```

# 五、API 边界

## 5.1 Patient-facing API

原则：

```text
只能返回患者安全表达。
不得泄露候选诊断列表。
不得泄露 must_not_miss 内部标签。
不得泄露 EvidenceGraph 内部推理。
不得泄露资产包内部调试信息。
```

## 5.2 Clinician-facing API

原则：

```text
可以返回候选诊断、证据状态、检查建议和医生摘要。
必须标注 evidence strength、uncertainty、capability boundary。
不得伪装成最终诊断。
```

## 5.3 Debug / Internal API

原则：

```text
路径必须包含 /api/v1/debug 或 /internal。
不得被 patient-facing client 调用。
Phase 5 前至少通过路径和文档隔离。
Phase 5 后接入权限控制。
```

## 5.4 Admin / Governance API

Phase 5 后置：

```text
Asset publish
Model registry
Evaluation management
Audit log
User / Role / Permission
```

# 六、脱敏策略

## 6.1 字段级脱敏

需要脱敏字段：

```text
姓名
手机号
身份证号
地址
医院号 / 病案号
具体就诊时间
医生姓名
机构名称
原始对话中的身份线索
```

## 6.2 RuntimeTrace 脱敏

策略：

```text
保存 raw_input_hash。
保存 sanitized_input_summary。
完整 raw_input 只在必要阶段短期保存。
导出 Trace 时默认使用 sanitized 版本。
```

## 6.3 EvaluationCase 脱敏

规则：

```text
优先使用合成病例。
真实病例必须去标识化。
ExpectedOutcome 不包含真实患者身份信息。
```

## 6.4 TrainingExampleCandidate 脱敏

规则：

```text
未脱敏样本不得进入 TrainingDatasetVersion。
必须记录脱敏方法。
必须保留来源 trace_id / evaluation_run_id 的可追踪引用，但不暴露原始患者身份。
```

# 七、日志与审计

## 7.1 日志禁止项

```text
不打印完整患者原始输入。
不打印完整 RuntimeState。
不打印完整身份证明信息。
不在 error log 中输出完整病例。
不在模型请求日志中输出未脱敏患者数据。
```

## 7.2 审计事件

Phase 5 应记录：

```text
Runtime viewed
Trace exported
Asset published
Asset rolled back
CapabilityProfile updated
ModelProvider approved
TrainingDataset created
EvaluationResult viewed
Patient-facing output generated
```

## 7.3 AuditLog 字段

```text
audit_id
actor_id
actor_role
action_type
resource_type
resource_id
time
ip / device
reason
before_version
after_version
```

# 八、训练数据安全

训练数据来源：

```text
EvaluationCaseSet
RuntimeTrace
Doctor Feedback
Follow-up Outcome
SafetyViolation
RegressionFinding
```

进入训练集前必须经过：

```text
1. 脱敏。
2. 样本质量检查。
3. 来源记录。
4. 风险标注。
5. 人工或规则审核。
6. TrainingDatasetVersion 版本化。
```

禁止：

```text
未审核 RuntimeTrace 自动进入训练集。
包含真实身份信息的样本进入模型训练。
模型训练后自动扩大患者端输出权限。
```

# 九、按 Phase 接入计划

## Phase 1–3

```text
通过路径和响应结构隔离 patient / clinician / debug。
不做完整权限系统。
不使用真实敏感病例。
不做训练数据自动沉淀。
```

## Phase 4

```text
引入脱敏后的 ExperienceCandidate / TrainingExampleCandidate。
明确 RuntimeTrace 作为复盘输入的脱敏要求。
```

## Phase 5

```text
Spring Security / JWT / RBAC。
AuditLog。
数据脱敏导出。
权限化 Console。
训练数据治理。
模型版本审核。
```

# 十、最终结论

ClinMindRuntime 在个人项目阶段不宣称临床合规落地，但必须体现医疗 AI 系统的安全边界意识。

核心结论：

```text
患者端看安全表达，
医生端看证据和不确定性，
后台端看 Trace 和治理信息，
训练数据必须脱敏、审核、版本化。
```
