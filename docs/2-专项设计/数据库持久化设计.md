# 数据库持久化设计

> 上位总设计：`docs/ClinMindRuntime完整系统设计.md`  
> 文档地图：`docs/00_项目设计地图.md`  
> 对应能力域 / 架构层：Storage / Integration 层；平台治理层；Persistence / Audit / Repository 治理底座  
> 当前状态：专项设计 / 已部分落地  
> 当前实现：Phase 5-P0 已实现 PostgreSQL 持久化、Repository 双实现、AuditLog、Persistence health / Audit API；pgvector、ModelRegistry、KnowledgeStore、TrainingDatasetVersion 正式发布链路仍后置  
> 对应 Phase：Phase 5-P0 已落地核心持久化；Phase 7/8/10 继续扩展向量检索、模型注册和生产级治理  
> 实现入口：持久化相关开发必须以对应 Phase 实现规格、数据库 Schema 设计和迁移任务清单为直接依据，不应仅凭本文档直接改表。

> 本文档定义 ClinMindRuntime 从 YAML / in-memory 走向 PostgreSQL / Redis / pgvector 的持久化演进方案。  
> 当前 Phase 1–3 不接数据库，本文档用于约束 Phase 4/5 之前的表结构方向、存储边界和迁移策略。

---

# 一、文档定位

当前项目使用：

```text
RuntimeStore：in-memory
AssetPackageRepository：YAML resources
EvaluationCaseRepository：YAML resources
EvaluationRunStore：in-memory
```

原因：

```text
Phase 1–3 重点是验证 Runtime、Asset、Evaluation 主线，不让数据库复杂度干扰 P0。
```

数据库持久化从 Phase 4/5 开始逐步引入。

---

# 二、技术选型

默认主数据库：

```text
PostgreSQL
```

原因：

```text
JSONB 支持好。
适合存储 Trace / Evaluation / Asset metadata。
可以通过 pgvector 支撑向量检索。
比单独引入 MySQL + Milvus 更适合当前项目阶段。
```

辅助组件：

```text
Redis：session cache / lock / rate limit / job progress。
pgvector：evidence / similar case / experience embeddings。
```

后置组件：

```text
Neo4j：复杂图遍历明确后再引入。
Milvus / Qdrant：向量规模和 QPS 超出 pgvector 后再引入。
```

---

# 三、持久化原则

```text
1. RuntimeState 可以持久化，但 Runtime 主控仍在 Java Runtime Core。
2. RuntimeTrace 必须可回放、可审计、可脱敏。
3. AssetPackage 必须版本化。
4. EvaluationResult 必须与 asset_package_version、runtime_version、case_set_version 绑定。
5. TrainingDatasetVersion 和 ModelProviderMetadata 必须可追踪来源。
6. 数据库不能让模型或 Provider 绕过 Runtime 边界。
```

# 七、迁移策略

## Phase 1–3

```text
不接数据库。
保留 in-memory / YAML。
```

## Phase 4

```text
可以先持久化 RuntimeTrace、EvaluationResult、ExperienceCandidate、TrainingExampleCandidate。
```

## Phase 5

```text
完整接入 PostgreSQL。
接入 Flyway / Liquibase 管理 schema。
接入 pgvector。
接入 Redis。
```

---

# 八、禁止边界

```text
1. 不在 Phase 3-P0 提前接 PostgreSQL。
2. 不为了“看起来完整”提前设计过细表结构。
3. 不让数据库触发器或存储过程承担 Runtime 主控。
4. 不让持久化层直接改变 CapabilityProfile。
5. 不把未脱敏 RuntimeTrace 直接写入训练数据表。
```

---

# 九、最终结论

数据库持久化的目标不是替代 Runtime，而是支撑审计、评估、经验、训练数据和平台治理。

路线：

```text
Phase 1–3：YAML + in-memory
Phase 4：关键复盘数据原型持久化
Phase 5：PostgreSQL + pgvector + Redis + 审计治理
```
