# 企业级智能诊断训练—运行—经验进化平台设计

## 一、项目定位

本项目是一个企业级智能诊断训练—运行—经验进化平台。

它不是普通医疗问答系统，也不是直接替代医生的全科 AI 医生，而是一个面向多症状群、多角色、多知识源、多评估闭环的智能诊断支持系统。

系统的核心目标是：让 AI 不只是调用医学知识回答问题，而是像受训医生一样，通过标准知识学习、病例训练、真实问诊、医生反馈、随访结局和经验复盘，逐步积累临床经验，并在安全边界内完成更自然、更动态、更接近医生思维的诊断推理。

一句话定义：

> 本项目希望把医疗 AI 从“一次性回答医学问题”，升级为一个能够训练、积累经验、动态推理、自然沟通、复盘再认证的企业级智能诊断 Runtime。

系统不是让大模型自由生成医学答案，而是让它在一个受控的诊断运行时中工作：它知道自己在哪些症状群中具备能力，知道当前病例有哪些候选诊断，知道哪些证据支持或反对某个方向，知道下一步应该追问什么或建议什么检查，也知道什么时候不能继续在线判断、必须建议就医或转医生。

------

## 二、设计核心：训练、经验、运行、表达、复盘

系统由五条主线组成：

```text
1. 训练：通过症状群 Rotation 和病例考试获得能力
2. 经验：通过 Clinical Experience Memory 积累临床经验
3. 运行：通过 Diagnostic Runtime 管理一次问诊推理
4. 表达：通过 Human-like Interaction Layer 让沟通更像医生
5. 复盘：通过医生反馈、随访结局和再认证持续进化
```

整体闭环：

```text
症状群 Rotation
  ↓
标准病例训练
  ↓
病例考试
  ↓
能力档案 Capability Profile
  ↓
真实问诊 / 医生协作 / Silent Evaluation
  ↓
CaseFrame + EvidenceGraph + DecisionBoundary 运行
  ↓
医生反馈 / 最终诊断 / 随访结局
  ↓
经验候选发现 Shadow Learning
  ↓
医生审核与质控
  ↓
Clinical Experience Memory
  ↓
再评估 / 再认证
  ↓
能力档案更新
```

这条闭环是本系统区别于普通医疗 Agent 的核心。

普通医疗 Agent 主要依赖：

```text
医学知识 + RAG + Prompt
```

本系统依赖的是：

```text
医学知识 + 病例经验 + 诊断状态 + 医生反馈 + 再认证
```

也就是说，系统不仅“知道医学知识”，还要逐步形成“遇到某类病例时应该警惕什么、优先问什么、不能轻易下什么结论”的临床经验。

压缩成一个核心公式：

```
智能诊断 = 医学知识 + 临床经验 + 诊断状态 + 输出边界 + 复盘进化
```

对应模块是：

```
医学知识：Clinical Pathway + KG-lite + RAG Evidence Library
临床经验：Clinical Experience Memory + Shadow Learning
诊断状态：CaseFrame + Differential Diagnosis Board + EvidenceGraph
输出边界：Capability Profile + SafetyGate + DecisionBoundary
复盘进化：Doctor Feedback + Follow-up Outcome + Recertification
```

------

## 三、系统要解决的问题

普通医疗大模型问诊不是完全没有医学知识，而是缺少稳定的诊断机制。

主要问题包括：

```text
1. 多轮问诊状态容易丢失
2. 追问容易机械化，缺少诊断目的
3. 候选诊断和证据关系不可追踪
4. 容易在信息不足时过度下结论
5. 不知道自己在哪些场景有能力、哪些场景没有能力
6. 缺少真实病例经验沉淀机制
7. 出错后无法像医生一样复盘和成长
8. 患者沟通容易模板化，要么过度自信，要么过度保守
```

本系统要解决的是：

> 如何让 AI 在明确能力范围内，像医生一样围绕诊断目标收集证据，像医生一样参考相似病例经验，像医生一样从错误和随访结果中成长，并且知道什么时候可以判断、什么时候必须转医生。

------

## 四、系统总体架构

系统由七个核心模块组成：

```text
1. Symptom Rotation Library：症状群训练库
2. Capability Profile：能力档案
3. Clinical Experience Memory：临床经验记忆
4. Diagnostic Runtime：诊断运行时
5. Shadow Learning：受控经验发现层
6. Human-like Interaction Layer：类医生交互层
7. Review & Recertification：复盘与再认证
```

运行流程：

```text
用户 / 医生输入病例
  ↓
CaseFrame 结构化病例
  ↓
识别症状群
  ↓
查询 Capability Profile
  ↓
检索 Clinical Experience Memory 中的相似经验
  ↓
进入 Diagnostic Runtime
  ↓
生成候选诊断与 EvidenceGraph
  ↓
进行动态追问或检查建议
  ↓
DecisionBoundary 控制输出等级
  ↓
Human-like Interaction Layer 生成自然表达
  ↓
医生反馈 / 最终诊断 / 随访结局
  ↓
Shadow Learning 发现经验候选
  ↓
医生审核后进入 Clinical Experience Memory
  ↓
再认证后更新能力档案
```

## 五、医生培养机制如何真正落地

医生培养机制不是文档中的一个章节，而是系统运行权限和经验进化的基础。

### 1. 症状群 Rotation：像医生轮转一样获得能力

系统不一开始承诺覆盖所有诊断问题，而是按症状群逐步训练和验证。

第一阶段可以覆盖五个高频症状群：

```text
发热
咳嗽 / 呼吸困难
腹痛 / 呕吐 / 腹泻
胸痛 / 胸闷
头痛 / 眩晕
```

每个症状群相当于一个 rotation。

每个 rotation 包含：

```text
标准医学知识
临床路径
常见候选诊断
必须排除的高危疾病
必问病史
推荐检查
典型病例
非典型病例
易误诊病例
医生点评
评估指标
```

例如胸痛 rotation 中，系统必须学习：

```text
常见方向：胃食管反流、肌肉骨骼痛、焦虑相关胸闷
高危方向：急性冠脉综合征、主动脉夹层、肺栓塞、张力性气胸
必问问题：放射痛、大汗、气短、活动后加重、下肢肿痛、近期制动
关键检查：心电图、肌钙蛋白、D-二聚体、胸部影像
```

这使系统不是机械套模板，而是在每个症状群内形成有边界的诊断能力。

------

### 2. 病例考试：决定 AI 获得什么权限

完成一个 rotation 后，系统必须通过病例考试，才能在该症状群上获得对应能力等级。

考试不只看最终诊断是否命中，而是看完整诊断行为：

```text
危险信号是否识别
高危疾病是否排除
必问问题是否覆盖
候选诊断是否合理
证据归因是否正确
推荐检查是否合理
是否过度自信
是否给出不安全建议
是否知道什么时候转人工
医生摘要是否可用
沟通是否清楚自然
```

能力档案示例：

```json
{
  "agent_id": "diagnosis_runtime_v1",
  "capability_profile": {
    "fever": {
      "level": "L3_candidate_diagnosis",
      "case_exposure": 520,
      "red_flag_recall": 0.95,
      "unsafe_output_rate": 0.02,
      "common_pitfalls": [
        "早期肺炎漏问呼吸情况",
        "持续高热低估脱水风险"
      ]
    },
    "chest_pain": {
      "level": "L2_triage_only",
      "case_exposure": 210,
      "red_flag_recall": 0.91,
      "unsafe_output_rate": 0.05,
      "common_pitfalls": [
        "年轻患者胸痛误判为焦虑",
        "未充分排除肺栓塞"
      ]
    }
  }
}
```

能力等级定义：

```text
L1：只能做病史结构化
L2：可以做危险信号识别和就医分级
L3：可以给候选诊断方向
L4：低风险、信息充分场景下给初步判断
L5：严格验证后的受控自主诊断
```

运行时系统会先查能力档案。
如果某个症状群只通过 L2，系统就不能输出候选诊断，只能做风险分诊和就医建议。

这就是医生培养机制的落地：

> 训练和考试结果直接决定系统运行时的诊断权限。

## 六、Clinical Experience Memory：让系统积累临床经验

医生的经验不是简单背更多知识，而是在大量病例中形成敏感性：

```text
哪些表现容易误导
哪些高危病早期很像普通病
哪些问题必须追问
哪些检查真正改变判断
哪些患者表达方式容易造成误判
哪些看似矛盾的信息提示病情正在变化
```

系统用 Clinical Experience Memory 沉淀这种经验。

它不是普通聊天记忆，也不是原始病例库，而是经过医生审核、质控和再认证的经验单元。

经验单元示例：

```json
{
  "experience_id": "chest_pain_exp_023",
  "symptom_group": "chest_pain",
  "case_pattern": {
    "initial_impression": "年轻患者胸痛，情绪紧张，容易被判断为焦虑",
    "hidden_risk": "肺栓塞",
    "trigger_features": [
      "胸痛",
      "气短",
      "近期久坐",
      "焦虑样表现"
    ]
  },
  "clinical_lesson": "年轻患者胸痛不能直接归因于焦虑。如果合并气短、近期久坐、下肢肿痛或激素/避孕药使用，需要优先排除肺栓塞。",
  "must_ask": [
    "是否呼吸困难",
    "是否咯血",
    "是否近期长途旅行或久坐",
    "是否下肢肿痛",
    "是否使用激素或避孕药"
  ],
  "recommended_tests": [
    "D-二聚体",
    "肺动脉CTA"
  ],
  "source": "doctor_reviewed_case",
  "validation_status": "approved_for_runtime_reference",
  "applicable_scope": "young_or_middle_aged_chest_pain_with_dyspnea",
  "not_applicable_when": [
    "明确外伤后胸痛",
    "已完成相关检查并排除肺栓塞"
  ]
}
```

Clinical Experience Memory 在运行时有四个作用：

```text
1. 相似病例提醒
当前病例与历史误诊或高风险模式相似时，触发经验提醒。

2. 调整候选诊断优先级
如果经验单元提示某类高危病容易伪装，系统会提高该候选诊断的排除优先级。

3. 影响追问策略
系统不再机械补字段，而是优先询问历史上最容易漏问、但最能改变判断的问题。

4. 修正输出边界
如果当前病例匹配高风险经验模式，即使表面症状不重，系统也不能轻易给低风险判断。
```

这使系统具备更接近医生的“经验感”。

------

## 七、Shadow Learning：受控无监督经验发现机制

系统不应该完全禁止无监督学习。

如果完全禁止，AI 很难像医生一样从大量病例中发现新的模式。但医疗场景也不能允许模型从真实病例中自动学习后直接改变临床判断。

因此，本系统采用“受控无监督经验发现机制”。

它分为三层。

### 1. Shadow Learning：影子学习层

系统可以在后台对真实病例进行无监督或自监督分析，例如：

```text
相似病例聚类
症状组合模式发现
高危病例早期表现挖掘
常见漏问问题统计
诊断修正路径分析
医生驳回原因聚类
随访结局与初始判断差异分析
```

这些学习结果只进入影子层，不直接影响线上输出。

### 2. Experience Candidate：经验候选层

系统从影子学习中生成经验候选。

例如：

```json
{
  "candidate_experience": "部分年轻胸痛患者在初始对话中表现为焦虑，但若合并气短和近期久坐，最终诊断中肺栓塞比例更高。",
  "supporting_cases": 18,
  "risk_level": "high",
  "suggested_runtime_action": "在年轻胸痛伴气短场景中优先追问久坐、下肢肿痛和咯血。",
  "status": "pending_review"
}
```

经验候选不会直接进入 Runtime，只会进入医生审核或质控流程。

### 3. Validated Experience：验证经验层

只有满足以下条件，经验才能进入 Clinical Experience Memory 并影响运行时：

```text
经过医生审核
有足够病例支持
明确适用范围
明确不适用范围
通过离线评估
不增加高危漏放率
版本记录完整
```

通过验证后，经验才可以影响：

```text
候选诊断优先级
追问策略
检查建议
风险提示
输出边界
```

这套机制使系统既能像医生一样从经验中成长，又不会让未经验证的模式直接污染临床输出。

## 八、Diagnostic Runtime：一次问诊如何推理

诊断 Runtime 负责一次具体问诊中的推理过程。

它由五个核心对象组成：

```text
CaseFrame
SafetyGate
Differential Diagnosis Board
EvidenceGraph
DecisionBoundary
```

------

### 1. CaseFrame：病例状态

CaseFrame 把用户或医生输入转换为结构化病例。

```json
{
  "patient_profile": {
    "age": 58,
    "sex": "male",
    "risk_factors": ["hypertension", "smoking"]
  },
  "chief_complaint": "胸闷胸痛",
  "present_illness": {
    "duration": "2小时",
    "trigger": "活动后加重",
    "associated_symptoms": ["出汗", "气短"]
  },
  "history": {
    "past_history": ["高血压"],
    "medication": [],
    "allergy": []
  },
  "missing_slots": [
    "心电图",
    "肌钙蛋白",
    "疼痛放射部位"
  ],
  "conflicting_slots": []
}
```

CaseFrame 让系统不依赖上下文记忆，而是显式维护病例状态。

------

### 2. SafetyGate：危险信号识别

SafetyGate 优先于诊断模块运行。

```json
{
  "risk_level": "high",
  "red_flags": [
    "胸痛",
    "出汗",
    "气短",
    "活动后加重"
  ],
  "allow_patient_facing_diagnosis": false,
  "next_action": "recommend_emergency_care"
}
```

如果 SafetyGate 判断高危，系统不能继续输出普通解释，而应建议急诊或转医生。

------

### 3. Differential Diagnosis Board：候选诊断池

系统不直接输出唯一诊断，而是维护候选诊断池。

```json
{
  "candidates": [
    {
      "name": "急性冠脉综合征",
      "risk_level": "high",
      "status": "must_not_miss"
    },
    {
      "name": "主动脉夹层",
      "risk_level": "high",
      "status": "need_to_rule_out"
    },
    {
      "name": "胃食管反流",
      "risk_level": "medium",
      "status": "possible_after_exclusion"
    }
  ]
}
```

高危候选不能因为置信度低就删除，只能标记为必须排除或需要排除。

------

### 4. EvidenceGraph：证据状态图

EvidenceGraph 记录每个候选诊断的证据状态。

```json
{
  "diagnosis": "急性冠脉综合征",
  "supporting_evidence": [
    "胸痛",
    "活动后加重",
    "出汗",
    "气短",
    "高血压史"
  ],
  "opposing_evidence": [],
  "missing_evidence": [
    "心电图",
    "肌钙蛋白",
    "疼痛放射部位"
  ],
  "experience_alerts": [
    "胸痛伴出汗和气短时，不能因症状短暂缓解而判断低风险"
  ],
  "recommended_questions": [
    "疼痛是否向左肩、后背或下颌放射？"
  ],
  "recommended_tests": [
    "心电图",
    "肌钙蛋白"
  ],
  "status": "need_to_rule_out"
}
```

EvidenceGraph 的作用：

```text
决定下一步问什么
决定是否建议检查
决定是否允许输出诊断方向
决定是否需要转急诊
决定哪些经验提醒需要触发
```

它不是解释层，而是诊断控制层。

------

### 5. DecisionBoundary：输出边界

DecisionBoundary 根据能力档案、风险等级、证据状态和经验提醒决定当前能输出什么。

输出等级：

```text
O1：继续追问
O2：风险提示
O3：候选诊断方向
O4：低风险初步判断
O5：门诊 / 急诊建议
O6：转人工医生
O7：医生端完整鉴别诊断报告
```

示例：

```json
{
  "allowed_output": "O5",
  "patient_facing_allowed": true,
  "clinician_copilot_allowed": true,
  "diagnostic_label_allowed_for_patient": false,
  "reason": [
    "当前为胸痛症状群",
    "系统能力等级为 L2，仅允许分诊",
    "存在高危心血管信号",
    "急性冠脉综合征尚未排除",
    "匹配胸痛高危经验单元"
  ]
}
```

系统不是“生成答案后加免责声明”，而是先判断当前是否允许生成。

------

## 九、Question / Test Policy：动态追问与检查建议

医生诊断不是机械问一串固定问题，而是根据当前候选诊断和缺失证据决定下一步。

系统的 Question / Test Policy 负责决定：

```text
下一步应该追问什么
是否需要建议检查
这个问题或检查服务于哪个候选诊断
它是在确认常见病，还是排除高危病
它是否能显著改变当前判断
```

优先级：

```text
危险信号
> 高危疾病排除
> 经验记忆提醒
> 高信息增益问题
> 关键检查建议
> 常见病鉴别
> 普通病史补全
```

示例：

```json
{
  "action_type": "ask_question",
  "content": "疼痛是否向左肩、后背或下颌放射？",
  "purpose": "rule_out_high_risk_diagnosis",
  "target_diagnoses": [
    "急性冠脉综合征",
    "主动脉夹层"
  ],
  "reason": "当前胸痛伴出汗、气短和活动后加重，需要优先排除高危心血管疾病。",
  "priority": "high"
}
```

检查建议示例：

```json
{
  "action_type": "recommend_test",
  "content": "建议尽快完善心电图和心肌损伤标志物检查。",
  "purpose": "rule_out_high_risk_diagnosis",
  "target_diagnoses": [
    "急性冠脉综合征"
  ],
  "priority": "urgent",
  "patient_facing_allowed": true,
  "clinician_mode_detail": "用于评估心肌缺血或损伤风险。"
}
```

这使系统不仅会“问问题”，还会“解释为什么问”，更接近医生诊断中的动态决策。

------

## 十、Human-like Interaction Layer：让系统更像真人医生

如果系统只输出结构化状态和规则，会显得机械。
因此系统增加 Human-like Interaction Layer。

它的作用不是改变医学判断，而是把 Runtime 的结果转化成更自然、更像医生的表达。

它有四个能力：

### 1. 解释为什么问

机械问法：

```text
有没有呼吸困难？
```

类医生表达：

```text
我需要先确认有没有呼吸困难，因为胸痛合并气短时，需要优先排除心肺方面的急症。你可以先告诉我，现在有没有喘不上气、说话费力或者嘴唇发紫？
```

### 2. 表达不确定性

机械判断：

```text
你这是胃病。
```

类医生表达：

```text
目前还不能先按胃部问题处理，因为胸痛、出汗和气短这些信息提示需要先排除心血管风险。我们先把最危险的情况排除掉，再看是否更像胃食管反流或肌肉疼痛。
```

### 3. 根据用户情绪调整语气

如果用户明显焦虑，系统先安抚，再说明下一步：

```text
我理解你现在会担心。我们先抓最重要的风险点，不需要一次想清楚所有问题。请先确认有没有胸痛加重、呼吸困难或大汗。
```

### 4. 用临床经验提醒风险

系统可以像医生一样提醒：

```text
有些胸痛一开始看起来像焦虑或胃部不适，但如果伴随出汗、气短或活动后加重，就不能先按低风险处理。
```

Human-like Interaction Layer 的边界：

```text
它只能改变表达方式
不能降低风险等级
不能改变 DecisionBoundary
不能编造临床经验
不能为了安抚用户而弱化就医建议
```

这样系统既不会机械，也不会为了像真人而失控。

------

## 十一、知识与证据系统

系统使用四类知识源：

```text
1. Clinical Pathway：症状群临床路径
2. KG-lite：结构化医学关系
3. RAG Evidence Library：指南、文献、说明书、科普资料
4. Clinical Experience Memory：审核后的临床经验
```

它们分工不同：

```text
Clinical Pathway：控制流程
KG-lite：提供疾病、症状、危险信号和检查关系
RAG：提供可追溯医学依据
Clinical Experience Memory：提供相似病例、误诊教训和经验提醒
```

核心原则：

```text
RAG 不直接决定诊断
知识图谱不单独完成诊断
经验记忆不自动覆盖指南
Shadow Learning 不直接影响线上输出
真正控制诊断的是 Runtime 状态
```

------

## 十二、企业级运行模式

系统提供三种模式。

### 1. Patient-facing Mode

面向患者，只允许输出：

```text
风险提示
就医建议
需要补充的信息
健康教育
需要携带的检查资料
```

禁止输出：

```text
确定诊断
处方
复杂检查结论
高危场景下的居家观察建议
```

### 2. Clinician Copilot Mode

面向医生，允许输出：

```text
候选诊断
证据状态图
相似病例经验提醒
推荐追问
推荐检查
医生摘要
鉴别诊断报告
```

AI 提供支持，医生负责最终判断。

### 3. Silent Evaluation Mode

在真实业务中，AI 生成判断但不影响医生决策。

系统记录：

```text
AI 候选诊断
AI 风险分级
AI 推荐检查
AI 经验提醒
医生判断
最终诊断
随访结果
```

用于评估系统真实表现和后续再认证。

------

## 十三、复盘与再认证

真实问诊后，医生反馈和随访结局进入复盘流程。

流程：

```text
真实病例
  ↓
医生反馈
  ↓
最终诊断 / 随访结果
  ↓
错误归因
  ↓
Shadow Learning 发现经验候选
  ↓
医生审核
  ↓
Clinical Experience Memory 更新
  ↓
离线评估
  ↓
再认证
  ↓
Capability Profile 更新
```

错误类型包括：

```text
主诉理解错误
危险信号漏问
候选诊断漏召回
证据归因错误
检查建议不合理
经验提醒误触发
输出越权
过度诊断
低估风险
沟通表达不清
```

如果新经验或新版本导致高危漏放率上升，系统应降低对应症状群权限，而不是自动升级能力。

这对应医生继续教育和再认证。

## 十四、评估体系

系统评估不是只看诊断命中率，而是看整个诊断行为是否安全、合理、像医生。

评估指标包括：

```text
危险信号识别率
高危病例漏放率
必问问题覆盖率
候选诊断 Top-3 召回率
推荐检查合理率
经验提醒命中率
不安全建议率
输出边界正确率
证据引用准确率
医生摘要可用性
沟通自然度
不确定性表达质量
医生反馈采纳率
```

医生 Rubric 评价维度：

```text
病史采集是否充分
是否抓住关键危险信号
候选诊断是否合理
追问是否有临床目的
检查建议是否合理
经验提醒是否有价值
表达是否自然但不过度承诺
是否适合给患者看
是否适合给医生参考
```

## 十五、参考对标与吸收点

本项目不是凭空设计，而是吸收了当前医疗 AI、诊断 Agent、医生 Copilot 和 Agent Runtime 方向中的关键思想。

### 1. AMIE：诊断对话不是闲聊，而是目标导向的信息采集

AMIE 的启发是：医疗 AI 的核心不是会回答医学知识，而是能在多轮对话中围绕诊断目标持续采集信息。

本项目吸收为：

```text
CaseFrame
Question / Test Policy
Human-like Interaction Layer
```

系统不是机械问字段，而是每一轮都知道：

```text
为什么问这个问题
它服务于哪个候选诊断
它是在确认常见病，还是排除高危病
```

### 2. 多模态诊断对话：真实医生不只看文本

多模态诊断对话的启发是：真实医生不仅听主诉，还看检查、图像、病历和病程变化。

本项目吸收为：

```text
多源输入
检查报告解析
影像 / 图片输入扩展
随访结果进入经验记忆
```

第一阶段可以先做文本和检查报告，后续扩展图片、心电、影像报告和病历文档。

### 3. MAI-DxO：诊断是顺序决策，不是一次性回答

MAI-DxO 的启发是：诊断过程需要不断决定下一步动作，是追问、请求检查，还是给出判断。

本项目吸收为：

```text
Question / Test Policy
Differential Diagnosis Board
EvidenceGraph
DecisionBoundary
```

系统不只是问问题，而是决定：

```text
下一步问什么
是否建议检查
这个检查的目的是什么
是否已经足够输出判断
```

### 4. HealthBench：医疗 AI 评估不能只看准确率

HealthBench 的启发是：医疗对话要用医生 rubric 评价，不能只看选择题或最终诊断命中。

本项目吸收为：

```text
医生 Rubric
行为级评估
沟通自然度评估
不确定性表达评估
输出边界正确率
```

系统评估的不只是“诊断对不对”，还包括：

```text
问得对不对
有没有漏危险信号
有没有过度确定
有没有误导患者
是否适合医生使用
```

### 5. OpenEvidence：医生需要可信证据，而不是黑箱答案

OpenEvidence 的启发是：医生端 AI 产品要提供可信的循证资料和可追溯依据。

本项目吸收为：

```text
RAG Evidence Library
证据引用
医生端 EvidenceGraph
知识源版本管理
```

系统面向医生时，不只输出答案，而是展示证据来源、证据关系、缺失信息和版本管理。

### 6. Hippocratic AI：患者侧医疗 Agent 必须有安全边界

Hippocratic AI 的启发是：患者侧医疗 Agent 要强调安全、升级机制和非诊断/非处方边界。

本项目吸收为：

```text
Patient-facing Mode 限权
SafetyGate
DecisionBoundary
人工升级机制
高危场景禁止普通建议
```

患者侧只给风险提示和就医建议，医生侧才展示完整候选诊断和证据状态。

### 7. 福棠·百川 / 小儿方：专科大模型和双医生制落地路线

福棠·百川的启发是：国内医疗 AI 落地更现实的路径是“真实医生 + AI 医生”的双医生制，而不是一开始让 AI 替代医生。

本项目吸收为：

```text
Clinician Copilot Mode
Silent Evaluation Mode
医生反馈闭环
能力档案与分级授权
```

本项目与其差异在于：

```text
福棠·百川更像儿科专科大模型落地产品。
本项目更像可扩展到多症状群、可积累临床经验、可再认证的企业级诊断 Runtime 平台。
```

### 8. DeepLearning.AI AI for Medicine：医学 AI 不只有诊断，还包括预测和治疗

AI for Medicine 的启发是：医学 AI 应拆成诊断、预后预测和治疗建议等不同任务。

本项目吸收为：

```text
第一阶段聚焦诊断支持
第二阶段扩展风险预测
第三阶段在医生端支持治疗方案参考
```

患者端不做治疗和处方，医生端只做辅助参考。

### 9. Stanford CS337：医疗 AI 要关注真实世界影响

真实医疗 AI 不能只做模型或 Demo，而要考虑真实医疗流程、用户、部署和影响。

本项目吸收为：

```text
Patient-facing Mode
Clinician Copilot Mode
Silent Evaluation Mode
审计日志
医生反馈
真实业务验证
```

系统从一开始就不是单一聊天界面，而是面向真实医疗工作流设计。

### 10. Berkeley LLM Agents：Agent 重点是状态、规划、工具、评估和人机协作

LLM Agent 的启发是：Agent 不是多套几个角色，而是要有状态、规划、工具调用、评估和人机协作。

本项目吸收为：

```text
State Orchestrator
CaseFrame
EvidenceGraph
Question / Test Policy
Evaluation Agent
Human-like Interaction Layer
```

系统的 Agent 不是堆数量，而是围绕诊断状态协同工作。

------

## 十六、核心亮点

### 1. Clinical Experience Memory：临床经验记忆

系统不仅有知识库，还有审核后的病例经验、误诊教训和医生点评，使 AI 能像医生一样从病例中积累经验。

### 2. 受控无监督经验发现

系统允许在 Shadow Learning 中进行无监督/自监督模式发现，但只有通过医生审核和再认证的经验才能影响 Runtime。

### 3. 医生培养机制进入 Runtime

训练、病例考试、能力档案、运行权限和再认证直接决定系统能输出什么。

### 4. EvidenceGraph 证据状态图

每个候选诊断都有支持证据、反对证据、缺失证据、经验提醒、推荐追问和推荐检查。

### 5. Human-like Interaction Layer

系统不是机械问诊，而是解释为什么问、表达不确定性、根据用户情绪调整语气，同时不改变医学安全判断。

### 6. DecisionBoundary 输出边界

系统先判断是否允许输出，再生成回答，避免信息不足或高危场景中越权回答。

### 7. 三种企业级运行模式

患者端、医生 Copilot、Silent Evaluation 分别服务患者沟通、医生辅助和真实验证。

------

## 十七、最终定义

本项目是一个企业级智能诊断训练—运行—经验进化平台。

它通过症状群 Rotation、病例考试、能力档案、Clinical Experience Memory、Shadow Learning、Diagnostic Runtime、Human-like Interaction Layer 和复盘再认证机制，让 AI 在安全边界内逐步积累临床经验，并在问诊过程中表现得更像一个受训医生：既能围绕诊断目标收集证据，也能根据相似病例经验提醒风险，还能用自然、有温度、不过度承诺的方式与患者沟通。

系统的核心价值不是“替代医生”，而是让医疗 AI 从普通问答升级为：

```text
可训练
可积累经验
可受控学习
可授权
可追踪证据
可动态决策
可自然沟通
可复盘再认证
```

的企业级智能诊断 Runtime。

# 补充模块：临床数据成熟度管线与经验记忆治理机制

## 一、Clinical Data Maturity Pipeline：临床数据成熟度管线

企业级智能诊断系统最大的现实问题之一是临床数据不足。

如果系统一开始就依赖真实医院病历、医生反馈和随访结局，会导致项目难以启动；但如果完全依赖大模型生成病例，又会造成系统只是在模拟数据上自我验证，缺少真实临床可信度。

因此，本系统采用“临床数据成熟度管线”，让数据来源从低风险、低成本、可启动的数据，逐步过渡到真实临床数据。

### 1. 数据分层

系统将数据分为五个成熟度等级：

```text
D0：医学知识数据
来源：指南、教材、临床路径、科普资料、药品说明书。
用途：构建 RAG Evidence Library、Clinical Pathway、KG-lite。
限制：只能提供医学依据，不能当作临床经验。

D1：标准教学病例
来源：公开病例、教材病例、医生手工编写病例。
用途：训练症状群 Rotation、构建病例考试。
限制：偏理想化，不能代表真实用户表达。

D2：模拟患者病例
来源：基于标准病例生成多轮问诊场景，包括表达模糊、焦虑、信息遗漏、前后矛盾等用户类型。
用途：训练问诊策略和 Human-like Interaction。
限制：只能训练流程能力，不能作为真实临床有效性证明。

D3：医生审核病例
来源：医生提供或审核后的脱敏病例。
用途：构建 Clinical Experience Memory，形成经验单元。
限制：需要标注来源、适用范围、审核状态和版本。

D4：真实业务病例与随访结局
来源：真实预问诊、医生反馈、最终诊断、检查结果和随访结果。
用途：Silent Evaluation、真实表现评估、经验进化。
限制：必须脱敏、合规、审计、医生审核，不能直接在线学习。
```

这五层数据对应系统的成长路径：

```text
D0 / D1：让系统具备基础医学知识和标准问诊能力
D2：让系统学会处理真实用户的非标准表达
D3：让系统开始积累医生审核后的临床经验
D4：让系统进入真实业务验证和持续进化
```

### 2. 数据用途隔离

不同数据不能混用。

系统将数据用途分为四类：

```text
1. Knowledge Data：知识数据
用于 RAG、KG-lite、Clinical Pathway。

2. Training Data：训练数据
用于症状群 Rotation、模拟患者、问诊策略优化。

3. Evaluation Data：评估数据
用于固定病例考试和对照实验，不能被训练阶段看到。

4. Experience Data：经验数据
用于 Clinical Experience Memory，必须来自医生审核或经过验证的病例复盘。
```

核心原则：

```text
训练集不能污染评估集。
模拟病例不能伪装成真实病例。
单个病例经验不能直接变成普遍规则。
未经审核的真实病例不能进入 Runtime。
```

这样可以避免系统在“看过的病例”上表现很好，但在真实病例中失效。

### 3. 冷启动方案

在没有真实临床数据的早期阶段，系统可以这样启动：

```text
第一步：用指南、教材、临床路径构建 D0 知识层。
第二步：用公开病例和医生编写病例构建 D1 标准病例集。
第三步：基于 D1 生成 D2 模拟患者多轮问诊数据。
第四步：用固定评估集对比 LLM-only、RAG-only 和本系统 Runtime。
第五步：在医生审核后，将部分高质量病例复盘转为 D3 经验单元。
第六步：后续进入 Silent Evaluation，逐步积累 D4 真实业务数据。
```

也就是说，系统不是一开始就要求大量真实病历，而是先用“标准知识 + 标准病例 + 模拟患者 + 小型评估”跑通核心机制，再逐步接入医生审核和真实反馈。

### 4. 数据缺口发现机制

系统应主动识别自己在哪些场景数据不足。

例如：

```text
某症状群高危病例太少
某类年龄段病例不足
某类主诉表达覆盖不足
某个候选诊断经常召回失败
某个危险信号经常漏问
某类检查建议经常被医生驳回
```

这些缺口会进入 Data Gap Queue。

```json
{
  "gap_id": "chest_pain_gap_012",
  "symptom_group": "chest_pain",
  "gap_type": "high_risk_case_insufficient",
  "description": "肺栓塞相关病例不足，系统在年轻胸痛伴气短场景下追问不稳定。",
  "suggested_action": [
    "补充医生审核病例",
    "生成模拟患者场景",
    "增加病例考试样本",
    "更新 Question/Test Policy"
  ]
}
```

这样系统不是被动等待数据，而是主动知道自己缺什么数据。

------

## 二、Experience Memory Governance：经验记忆治理机制

Clinical Experience Memory 是系统最有价值的模块，但也是最大风险来源。

如果经验记忆缺少治理，系统可能会出现：

```text
错误经验被固化
个别医生习惯被放大
少数病例被当作普遍规律
过期经验继续影响判断
经验与指南冲突
经验触发场景过宽
系统过度类比相似病例
```

因此，系统必须设计经验记忆治理机制。

### 1. 经验进入 Runtime 前必须经过四级状态

经验记忆不允许从病例直接进入 Runtime。

每条经验必须经过四个状态：

```text
Candidate：经验候选
由 Shadow Learning、医生反馈或错误复盘生成，不能影响线上输出。

Reviewed：医生已审核
医生确认这条经验有临床意义，但还不能直接影响 Runtime。

Validated：已验证经验
通过离线评估，证明不会增加高危漏放率或不安全建议率，可以进入 Runtime。

Deprecated：废弃或降权经验
过期、冲突、表现不稳定或被医生否定后，不再影响 Runtime。
```

经验状态示例：

```json
{
  "experience_id": "chest_pain_exp_023",
  "status": "validated",
  "source": "doctor_reviewed_case_cluster",
  "reviewer_role": "cardiology_physician",
  "evidence_level": "multi_case_pattern",
  "last_reviewed_at": "2026-06-01",
  "expires_at": "2027-06-01"
}
```

### 2. 经验必须有适用边界

每条经验都必须写明：

```text
适用于什么场景
不适用于什么场景
证据来源是什么
是否和指南一致
是否只适合医生端展示
是否允许影响患者侧输出
```

示例：

```json
{
  "clinical_lesson": "年轻患者胸痛不能直接归因于焦虑，若合并气短和近期久坐，需要排除肺栓塞。",
  "applicable_scope": [
    "young_or_middle_aged_patient",
    "chest_pain",
    "dyspnea_or_recent_immobility"
  ],
  "not_applicable_when": [
    "clear_trauma_related_chest_pain",
    "pulmonary_embolism_already_excluded",
    "symptoms_fully_explained_by_confirmed_non_urgent_condition"
  ],
  "allowed_runtime_effect": [
    "increase_question_priority",
    "add_experience_alert",
    "raise_triage_attention"
  ],
  "forbidden_runtime_effect": [
    "direct_final_diagnosis",
    "override_guideline",
    "lower_safety_level"
  ]
}
```

这样可以防止系统把一个局部经验泛化到所有病例。

### 3. 经验只能影响“提示和优先级”，不能直接决定诊断

Clinical Experience Memory 的作用是提醒系统：

```text
这个病例像某类历史误诊模式
这个问题过去容易漏问
这个高危方向需要优先排除
这个检查可能改变判断
```

但它不能直接说：

```text
因为相似病例是肺栓塞，所以当前病例就是肺栓塞。
```

经验记忆允许影响：

```text
候选诊断优先级
追问顺序
检查建议
风险提醒
输出边界收紧
```

不允许直接影响：

```text
最终诊断结论
处方建议
高危降级
跳过必要检查
覆盖指南原则
```

经验记忆的原则是：

> 经验只能让系统更警觉，不能让系统更武断。

### 4. 经验与指南冲突时，指南优先

系统知识优先级为：

```text
安全规则
> 用户明确事实
> 当前权威指南 / 临床路径
> 医生审核经验
> 相似病例
> LLM 假设
```

如果经验记忆与指南冲突：

```text
不能自动使用经验。
必须标记 conflict。
进入医生审核。
在患者侧不展示。
在医生端只作为“经验性提醒”展示。
```

冲突示例：

```json
{
  "conflict_id": "conflict_018",
  "experience_id": "fever_exp_041",
  "conflict_with": "current_clinical_pathway",
  "conflict_type": "triage_recommendation_conflict",
  "runtime_action": "do_not_apply_to_patient_facing_mode",
  "review_status": "pending_specialist_review"
}
```

### 5. 经验上线前必须通过回归测试

每次新增经验、修改经验或调整经验权重，都必须跑病例回归测试。

重点观察：

```text
高危病例漏放率是否上升
不安全建议率是否上升
过度诊断率是否上升
输出边界正确率是否下降
医生摘要是否变差
追问轮数是否异常增加
```

如果任何核心安全指标变差，该经验不能上线，或者只能在医生端以低权重提示。

### 6. 经验需要版本管理和回滚

Clinical Experience Memory 必须像代码一样管理版本。

每次变更记录：

```text
新增了哪些经验
废弃了哪些经验
调整了哪些适用范围
改变了哪些权重
影响了哪些症状群
评估结果是否通过
```

如果上线后发现某条经验导致误导，系统需要支持：

```text
单条经验下线
某个症状群经验回滚
某个版本整体回滚
自动降低相关能力等级
```

### 7. 经验污染监控

系统需要持续监控经验记忆是否产生污染。

污染信号包括：

```text
某条经验被异常频繁触发
某类经验导致系统过度诊断
某个医生来源的经验被大量驳回
某个症状群经验与指南冲突增多
经验触发后医生采纳率持续下降
经验触发后不安全建议率上升
```

对应处理：

```text
自动降权
暂停触发
进入人工复审
回滚版本
降低症状群能力等级
```

------

## 三、两个问题如何进入系统主线

加入这两个模块后，系统主线变成：

```text
症状群 Rotation
→ 标准病例训练
→ 病例考试
→ 能力档案
→ 真实问诊 / Silent Evaluation
→ Shadow Learning 发现经验候选
→ Experience Memory Governance 审核治理
→ Clinical Experience Memory
→ Runtime 使用经验提醒
→ 回归测试与再认证
→ 能力档案更新
```

也就是说：

```text
Clinical Data Maturity Pipeline 解决系统“怎么获得可靠经验”。
Experience Memory Governance 解决系统“怎么不被错误经验污染”。
```

这两个模块分别对应：

```text
临床数据不足 → 用数据成熟度管线逐步过渡
经验记忆污染 → 用经验治理、审核、适用边界、回归测试和回滚控制风险
```
